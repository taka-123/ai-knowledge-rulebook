## 目的

- 既存の正本構造、同期フロー、設定互換性、セキュリティを壊さずに、依頼された変更を最小範囲で完了する。
- 迷った場合は、推測よりも既存コード、設定ファイル、テスト、ドキュメントを優先する。

## プロジェクト概要

- 概要: AI エージェント設定、運用ルール、学習ノートの正本を管理するリポジトリ。
- 主要技術: Markdown、JSON / JSONC、YAML、Node.js 検証スクリプト、Shell 同期スクリプト。
- ランタイム: Node.js 18 以上。YAML / JSON Schema 検証に Python 系 CLI も使う。
- パッケージマネージャ: npm（`package-lock.json`）。
- 主要エントリポイント: `ai/`、`.claude/skills/`、`scripts/`、`schemas/`、`notes/`。
- 非自明な設計・制約: 正本は `ai/` 配下。ホーム配下のツール設定は配布先として扱い、同期スクリプトで反映する。

## 参照先

- 全体概要: `README.md`
- 技術スタック: `package.json`
- ディレクトリ構造: `README.md`

実態とドキュメントが食い違う場合は、実ディレクトリ、設定ファイル、テスト、依存関係マニフェストを優先し、必要に応じてドキュメント更新を提案する。

## 主要コマンド

- Install: `npm install`、`pip install check-jsonschema yamllint`
- Lint: `npm run lint`
- Test: `npm run schema:check`、`npm run agent:check`
- その他: `./scripts/sync-all-to-home.sh`、必要に応じて `./scripts/sync-*-to-home.sh`

## プロジェクト構造

- `ai/`: 各 AI ツール向け設定・ルールの正本。
- `.claude/skills/`: このリポジトリ内で使う Skill 定義。
- `scripts/`: `ai/` からホーム配下へ同期するスクリプトと検証スクリプト。
- `schemas/`: `ai/` と `notes/` 向け JSON Schema。
- `notes/`: 学習ノート。
- `snippets/`: テンプレート / 雛形。
- `.work/`: 作業用ドキュメント。
- 生成物: `node_modules/` や検証・出力用の一時ディレクトリは編集対象にしない。
- 直接編集してはいけないもの: `~/.claude`、`~/.cursor`、`~/.codex`、`~/.gemini`、`~/.codeium` などの配布先。

## 作業原則

- 既存アーキテクチャ、命名、責務分離、エラーハンドリング方針を優先する。
- 変更は、依頼を満たす最小範囲に留める。
- 既存の検証スクリプト、同期スクリプト、schema、設定を再利用する。
- 依頼範囲外のリファクタ、整形、抽象化、ファイル作成を行わない。
- 生成ファイルは直接編集せず、元ソースまたは生成手順を更新する。
- 新しい本番依存を追加・削除・更新する場合は、事前に承認を取る。
- 秘密情報、トークン、認証情報、環境固有値を作成・推測・出力しない。

## 事前探索と承認

- 実装前は `rg` / `grep` などで既存パターンを探索し、影響範囲（型、API、設定）を把握する。
- **承認または明示依頼がない限り行わない**: 依頼範囲外の変更、復元困難な削除、外部契約や公開APIに影響する変更、要件外リファクタ。

## タスク契約

非自明な変更では、実装前に次を短く整理する。

1. Goal — 何を変えるか
2. Scope — 触る可能性が高いファイル、モジュール、API、画面
3. Constraints — 互換性、設計、セキュリティ、性能、スタイル上の制約
4. Done when — 何を満たせば完了か

誤った解釈のコストが高い場合は確認する。そうでなければ、前提を明記して進める。

## プロジェクト固有ルール

- `ai/` を正本として編集し、ホーム配下への反映は `scripts/sync-*-to-home.sh` を使う。
- `.claude/skills/<name>/SKILL.md` が Skill 本文の置き場。ルーターや補助設定に本文を複製しない。
- Agent / Skill の description と Trigger Keywords は自動マッチに使われるため、責務や対象パスを具体的に書く。
- JSON、YAML、Markdown は既存の formatter / linter / schema に合わせ、検証を弱めて通さない。

## 検証

- 変更箇所に対応する最小のテストから実行する。
- 共有ルール、Agent、Skill、schema、同期スクリプトに触れた場合は、対応する `*:check` を実行する。
- Markdown、JSON、YAML を変更した場合は、必要に応じて lint または format check を実行する。
- 検証できない場合は、未検証の理由と、ユーザーが実行すべきコマンドを示す。

## 禁止事項

- 明示依頼なしに、同期先契約、CI/CD、依存関係を変更しない。
- テストを通すために、セキュリティチェック、型チェック、lint、失敗テストを無効化しない。
- 失敗を隠すための暫定コード、ダミー値、不要な TODO、デバッグ出力、コメントアウトされたコードを残さない。
- `dist/`、`build/`、生成コードは原則として直接編集しない。lockfile を変更する場合は、その理由を明示する。
- commit、reset、rebase、force-push、branch削除、PR作成は、明示依頼なしに行わない。

## 外部情報

外部SDK、フレームワーク、クラウドサービス、CLI、API仕様、バージョン依存の挙動に関わる場合は、内部知識だけで判断しない。

- まず、このリポジトリの `package.json`、lockfile、設定、import、既存使用例を確認する。
- 必要に応じて公式ドキュメント、公式リポジトリ、changelog、型定義を確認する。
- 外部情報を根拠にした場合は、参照元または確認した対象を簡潔に示す。

## Shell

- Shell実行時は、原則として明示的な作業ディレクトリを指定する。
- `cd` を前提にしたコマンド連鎖は避ける。
- 一時ファイルを作成した場合は、作業完了時に削除するか、削除提案を行う。

## 完了報告

完了時は、次を簡潔に報告する。

- 変更内容
- 変更ファイル
- 実行した検証と結果
- 未検証事項
- 残リスクまたは次に取るべき行動

## Review guidelines

PR・差分レビューでは `REVIEW.md` を確認し、そのレビュー方針に従う。
通常の実装作業では、レビューが必要な場合を除き `REVIEW.md` を読む必要はない。
