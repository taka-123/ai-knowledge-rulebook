# Codex Review Playbook

## 設計方針: コンテキストベース自動ルーティング

発話で「ファスト/ヘビー」を選ばせるのではなく、**差分量・ファイル数・リスク領域から AI が自動でルートを決める**。2024–2026 のトレンド（Context-based automatic routing、大規模 diff は並列エージェント）に合わせている。根拠: パワーユーザー向け AI コードレビュー調査で、トリガー文言よりシステムが深さを自動調整する方式が推奨され、大規模 diff の並列レビューが一般化している。

- ユーザーは「レビューして」「レビューしてください」のどちらでもよい。**同じ導線**。
- 必要なら明示オーバーライド: 「しっかりレビューして」→ パイプライン強制、「軽く見て」→ 単一パス強制。

---

## ルーティング判断（自動）

1. **まず差分を取得**: `git diff --stat` と `git diff -p` でスコープを把握する。
2. **ヒューリスティックで判定**:

| 条件                                                                                                                                         | ルート                                                                                           | 理由                                             |
| -------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ | ------------------------------------------------ |
| 差分が**小さい**（目安: 約 400 LOC 未満 かつ ファイル数 少なめ）かつ **リスク領域に触れていない**                                            | **単一パス**: ai-diff-review 手順を現コンテキストで実行（または 1 サブエージェントでスキル実行） | 時間・トークン節約、メインコンテキストを汚さない |
| 差分が**大きい**（約 400 LOC 以上 または 複数ファイル・多ファイル） または **リスク領域**（auth / security / infra / 依存更新 等）に変更あり | **パイプライン**: cross-service-reviewer → code-reviewer →（必要時）security-reviewer → verifier | 並列・多観点で高精度、フォールバックあり         |
| ユーザーが「しっかり」「full review」等で**明示**                                                                                            | 上記に関係なく **パイプライン**                                                                  | オーバーライド                                   |
| ユーザーが「軽く」「quick」等で**明示**                                                                                                      | 上記に関係なく **単一パス**                                                                      | オーバーライド                                   |

3. **requirements.md + 直近コミット** が依頼に含まれる場合は従来どおりパイプラインを推奨（要件駆動の厳密レビュー向け）。

---

## 使い方（ユーザー視点）

- **通常**: 「レビューして」「レビューしてください」「この修正でOKか」等、**発話を覚え分け不要**。システムが diff を見て単一パスかパイプラインかを決める。
- **小さい差分**: 単一パス（ai-diff-review）で速く結果が出る。
- **大きい差分**: 自動でパイプラインになり、複数観点・並列でレビューされる。

どちらのルートでも **出力形式は統一**（下記「統一レポート形式」）。

---

## 統一レポート形式（全ルート共通）

どのルートを通っても、ユーザー向けレポートは次の形式に揃える（絵文字付き）。詳細は ai-diff-review の Output Contract を正とする。

- **レビューサマリー**: Diff モード/Scope、リスクレベル（✅ None / 🟢 Low / 🟡 Medium / 🟠 High / 🔴 Critical）、概要
- **指摘事項**: 各項目に カテゴリ（Correctness/Security/Performance/Maintainability）、深刻度（🔴 High / 🟡 Medium / 🟢 Low）、該当箇所・説明・推奨対応・確信度

パイプライン通過時は、その後に「パイプライン状況」（Delegation / Fallback）を追記してよい。

---

## Natural Language Routing（起動条件）

- 「レビューして」「レビューしてください」「この修正でOKか」「見てほしい」等、**いずれの自然文でも**レビュー導線が起動する。
- 起動後は上記「ルーティング判断」に従い、**コンテキストで単一パス vs パイプラインを決定**する。
- `requirements.md` と `直近コミット` が両方含まれる場合、固定順序は以下:
  1. `cross-service-reviewer`
  2. `code-reviewer`
  3. `security-reviewer`（セキュリティリスクがある場合）
  4. `verifier`

## Fallback Rule

- `code-reviewer` または `security-reviewer` が失敗・タイムアウトした場合、`cross-service-reviewer` が不足観点を代替実施する。
- 最終レポートには必ず次を含める:
  - 失敗したサブエージェント名
  - 親が代替した観点

## Compatibility Gate

- no-stat 系の非互換オプションは禁止。
- 差分確認は `git diff --stat` と `git diff -p` を使う。
