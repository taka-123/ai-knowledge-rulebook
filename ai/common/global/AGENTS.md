# AI 共通グローバルルール

> Motto: "最小・明瞭・安全な自律解決"

## 1. 適用順序

- 作業対象から親へ遡って、ツール固有ルールと `AGENTS.md` を統合する。
- 競合時は作業パスに近いルールを優先し、最後に本ファイルを適用する。
- 共有ソースは必要時のみ参照し、未存在・不整合時は manifest と実ディレクトリを正とする。

## 2. 実行プロトコル

- Read-First: 実装前に関連ファイルと関連ドキュメントを確認する。
- 1ターンで `Plan -> Implement -> Verify` を完了する。
- 応答は原則 `Plan -> 根拠 -> 最小例 -> トレードオフ`。
- 不確実性が 20% 以上なら A/B/C 案と推奨案を併記する。
- 予期せぬエラー、計画変更、セキュリティリスクは即時報告する。

## 3. 情報鮮度と出典

- 最新性が絡む事項は内部知識のみで断定しない。
- `context7` を優先し、不足分を Web 検索で補完する（時制クエリを使う）。
- 出典優先順位は「公式ドキュメント > 公式リポジトリ > 信頼できる二次情報」。
- URL と日付を明記し、事実と推測を分離する。

## 4. セキュリティと実行制約

- `.env` や認証情報は閲覧・編集・出力しない。必要時は `<API_KEY>` を使う。
- 一時ファイル（`tmp_*` など）は完了時に削除または削除提案する。
- パイプ、リダイレクト、コマンド置換で禁止事項を回避しない。
- `/` コマンドは定義どおりのみ実行し、定義ファイルは生成・更新しない。
- `cd` / `pushd` / `popd` は使わず、`working_directory` または絶対パスを使う。

## 5. 実装・出力スタイル

- Markdown: 本文の連続バッククォート最大長を `n` とし、外側フェンスは `n+1` 本以上にする。
- Markdown: `diff` は前後 3 行以上のコンテキストを含める。
- Markdown: フェンス衝突、閉じ忘れ、不可視文字混入を防ぐ。
- 実装: 既存のインデント、命名、コメント密度を優先し、作業痕跡コメントは残さない。
- 実装: `rg` などで類似処理を探索し、重複実装を避ける。
- 実装: 1ファイル 300 行超が見込まれる場合は分割を検討する。

## 6. タスク運用

- 🟢 軽量: 10行以内の修正、調査、閲覧。
- 🟡 標準: 機能改修、API 連携、リファクタ。
- 🔴 重要: インフラ、DB スキーマ、セキュリティ、破壊的変更（段階実行・ロールバック前提）。
- 障害時: 🟢は記録、🟡は1回再試行、🔴は停止して報告する。
- 複数ステップでは `**進行中**: [✅✅⏳⬜] 2/4 完了` 形式で進捗共有する。
