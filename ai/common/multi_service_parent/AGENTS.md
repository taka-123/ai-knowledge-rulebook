- 横断作業で、対象リポジトリ、変更順序、契約境界、検証範囲を明確にする。
- 子リポジトリ固有の実装ルールは、各子リポジトリの `AGENTS.md` を優先する。
- リポジトリ名やディレクトリ名だけで責務を決めつけず、実ファイル、設定、依存関係、README、子リポジトリの `AGENTS.md` から実態を確認する。

## 子リポジトリの読み込み

- ユーザーが子リポジトリ名、サービス名、パス、機能名を言及した場合は、**早い段階で**該当する子リポジトリの `AGENTS.md` を読み、コンテキストを合わせる。
- **パスやリポジトリ名が明示されているときはそれを最優先**し、不明なときだけ最小限の探索で候補リポジトリを特定する。
- 子リポジトリの `AGENTS.md` と本ファイルが競合する場合は、子リポジトリ側を優先する。
- 子リポジトリに `AGENTS.md` がない場合は、README、設定ファイル、テスト、CI、既存コードから規約を推定し、推定であることを明記する。

## 横断タスク契約

複数リポジトリにまたがる非自明な変更では、実装前に次を短く整理する。

1. Goal — 何を達成するか
2. Affected repos — 影響するリポジトリと理由
3. Boundaries — API、DB、schema、認証、イベント、環境変数などの契約境界
4. Change order — 変更順序、互換性維持、移行手順
5. Done when — 各リポジトリおよび統合観点で何を満たせば完了か

誤ったリポジトリ選定や変更順序のコストが高い場合は確認する。そうでなければ、前提を明記して進める。

## 横断変更の原則

- 変更は、依頼を満たす必要最小限のリポジトリ・ファイル範囲に留める。
- 一方のリポジトリだけを変更して、API、schema、型、環境変数、認証、権限、migration、CIとの整合性を崩さない。
- 互換性を壊す変更は、段階移行、後方互換、ロールバック方法を検討する。
- 横断的なリファクタ、構成変更、依存更新、CI/CD変更は、明示依頼がない限り提案止まりにする。

## Shell

- Shell実行時は、原則として明示的な作業ディレクトリを指定する。
- `cd` を前提にしたコマンド連鎖は避ける。
- 一時ファイルを作成した場合は、作業完了時に削除するか、削除提案を行う。

## 完了報告

横断作業の完了時は、次を簡潔に報告する。

- 変更したリポジトリとファイル
- リポジトリごとの検証結果
- 横断的な整合性確認結果
- 未検証事項
- 残リスクまたは次に取るべき行動
