## 基本方針

- レビューコメントは日本語で書く。
- 識別子、設定キー、コード、ログ、引用は原文を維持する。
- 差分によって新たに入った問題、または差分で悪化した問題を優先して指摘する。
- 実害、根拠、最小修正案を示せる場合だけ指摘する。
- 同一原因の指摘は集約する。
- 秘密値、トークン、個人情報、内部情報は再掲せず伏せる。
- P0 / P1 相当の指摘がない場合は `指摘なし` と返す。

## レビュー観点

- セキュリティ: 秘密情報、トークン、個人情報、MCP 認証情報、`.env*` が差分に含まれていないか。
- 権限・認証・認可: `--include-mcp`、外部送信、ホーム配下同期、Git 操作などの副作用が明示的な承認範囲を超えていないか。
- 正確性・仕様整合: `ai/` を正本とする構造、同期先契約、`AGENTS.md` / `CLAUDE.md` の運用憲法としての責務と矛盾していないか。
- API互換性: 各ツールの設定形式、Agent / Skill frontmatter、Codex wiring、Cursor agent schema、既存コマンド名を壊していないか。
- データ整合性: JSON / YAML / Markdown / TOML が schema、formatter、既存の命名・参照関係と整合しているか。
- エラーハンドリング: 同期・検証スクリプトが失敗を握りつぶさず、退避、スキップ、警告の条件を保っているか。
- テスト: 変更対象に応じて `npm run lint`、`npm run schema:check`、`npm run agent:check` または個別 check が必要か。
- ドキュメント: README、scripts README、テンプレート、ルール本文が実ディレクトリ・設定・検証コマンドと食い違っていないか。
- 運用・保守性: 正本と配布先の境界、Skill 本文の重複禁止、最小差分、既存パターン優先が守られているか。
- 実行品質（実装・テスト変更時）: テストが仕様の翻訳になっているか。`TEST_CONTRACT.md` を正本として否定・S1–S8 が守られているか。TRACEABILITY は監査成果物として生成されているか。
- リポジトリ固有の内容ポリシー: 毎ターン参照されるルールには、高頻度判断に直結しない補足や長い手順を増やしていないか。
- リポジトリ固有の公開範囲・読者区分: グローバル配布される内容に、個人環境専用の前提や未検証の外部仕様を混ぜていないか。

## 優先度

1. セキュリティ・秘密情報・外部副作用・配布先破壊の防止
2. `ai/` 正本、同期スクリプト、schema、Agent / Skill 検証との整合
3. ツール別設定形式、frontmatter、routing、Codex wiring の互換性
4. 運用憲法としての簡潔さ、保守性、ドキュメント整合性

## 重大度

Codex GitHub Review では、原則として P0 / P1 相当の問題だけを指摘する。
P2 / P3 は、ローカルレビューや人間向けレビューで必要な場合だけ使用する。

- `P0`: 秘密情報の漏えい、認証情報を含む同期、ホーム配下や Git 履歴への破壊的影響、広範囲に誤った AI ルールを配布する問題
- `P1`: 正本・同期先契約・schema・Agent / Skill 検証を壊す問題、CI で防げるはずの重大な検証漏れ、主要ツールで設定が読めなくなる問題
- `P2`: 保守性低下、参照関係の不整合、冗長なルール追加、将来の同期・検証ミスにつながる問題
- `P3`: 表記ゆれ、軽微な読みやすさ、修正必須ではない文体・体裁の改善

## コメント形式

指摘には、可能な限り次を含める。

- 問題点
- 影響
- 根拠
- 最小修正案

形式:

```text
[P1] タイトル

問題:
影響:
根拠:
最小修正案:
```

- 未確認事項は `確認事項:` として分ける。
- 差分外の提案は `follow-up:` として分ける。
- 修正必須ではない好みの指摘は、その旨を明記する。
- 同じ修正で解消できる指摘は、複数コメントに分散させず集約する。

## 指摘しないもの

- 差分外の既存問題: 今回の変更で悪化していないものは、必要なら `follow-up:` に分ける。
- 自動生成物: `node_modules/`、検証出力、一時出力など、正本ではない生成物の直接修正要求。
- 既知の移行中コード: 明示された移行範囲内で、検証や同期契約を壊していない一時的な差分。
- スタイル上の好みだけの指摘: 既存 formatter / linter / schema に反しない表現の好み。
- P0 / P1 に該当しない軽微な体裁指摘: Codex GitHub Review では原則コメントしない。
