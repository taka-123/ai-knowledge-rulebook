# 検証ハーネス（VERIFICATION）

非自明な機能・自律実行向けの運用ガイド。  
毎ターン読む `AGENTS.md` / `CLAUDE.md` には要約のみ載せ、詳細は本書を**必要時に**読む。

否定テスト必須・S1–S8・実装側義務の**正本は `TEST_CONTRACT.md`**（テンプレ `TEST_CONTRACT.template.md`）。

## なぜ必要か

| 典型失敗                         | 原因                                                |
| -------------------------------- | --------------------------------------------------- |
| ルール破綻（ショートカット実装） | 実装が都合で仕様を上書き。テストも共謀              |
| 横展開漏れ                       | 1 箇所直したが類似機能へ未適用                      |
| 演出・体験欠落                   | 内部状態だけ確認。可視未 assert（可聴は別途委譲）   |
| 早すぎる完了報告                 | DoD（数えやすい条件）だけ満たし、上位品質を見落とす |

**下位 DoD を満たしただけでは、上位完成度（SPEC / AGENTS にあれば）を満たしたとは言えない** — これを構造で防ぐ。

## 単一エージェントの最小規律

役割分離なしでも、各マイルストーンで次を守る。

1. 実装前: 要件を肯定 / 否定 / 不変条件に翻訳し進捗ログに残す
2. `TEST_CONTRACT.md` に従う（否定・S1–S8・実装容易性）
3. 各検証に「実装が壊れても通るか？」を自問。Yes なら否定を足す
4. UI ありの場合は**可視**まで確認（可聴は自動化不能欄へ）
5. 固定 sleep / skip / 閾値緩和 / 正解集合を狭める補正は禁止
6. UI 変更は可能ならスクリーンショット目視

## 役割分離（必要時のみ）

並列のためではなく、**共謀防止・第三者監査**のために使う。

| 役割         | タイミング                 | やること                         | 禁止                                 |
| ------------ | -------------------------- | -------------------------------- | ------------------------------------ |
| verify-grill | 実装**前**                 | 肯定・否定の検証設計             | 実装・テストコードを書かない         |
| implementer  | 設計**後**                 | 仕様 + TEST_CONTRACT どおり実装  | テストを変更しない                   |
| test-author  | 実装**後**                 | 敵対的テスト追加                 | 実装を曲げて通すこと                 |
| spec-auditor | 検証**後**・完了報告**前** | TRACEABILITY 生成 + 不変条件監査 | テストが通っただけで PASS とすること |

### いつ分離するか

| 状況                                            | 推奨                                 |
| ----------------------------------------------- | ------------------------------------ |
| 自己規律が信頼できるモデル + TEST_CONTRACT あり | 上記「最小規律」で単一エージェント可 |
| テストと実装の共謀が起きた                      | test-author を**別会話**で起動       |
| 完了判定                                        | spec-auditor（文脈非共有）           |

## 推奨フロー（マイルストーン / 機能単位）

```
1. verify-grill   … 肯定/否定一覧（進捗ログに残す）
2. implementer    … TEST_CONTRACT を読み実装
3. test-author    … 敵対的テスト（失敗していてよい）
4. implementer    … 本質修正。回帰も通す
5. spec-auditor   … TRACEABILITY 生成 + 不変条件。監査が FAIL なら差し戻し（同一不具合 3 回で停止）
6. スクリーンショット目視（UI ありの場合）
```

## 関連ファイル（プロジェクト側）

| ファイル                     | 用途                                                                |
| ---------------------------- | ------------------------------------------------------------------- |
| `TEST_CONTRACT.md`           | **正本**。テンプレからコピー + `rulebook 版:` 行                    |
| `INCIDENT_CATALOG.md`        | 任意。S1–S8 の経験的根拠（テンプレ `INCIDENT_CATALOG.template.md`） |
| `docs/audits/TRACEABILITY-*` | 監査レポート本文を**呼び出し側**が保存（spec-auditor は read-only） |
| `docs/VERIFICATION.md`       | 本書のコピーは**任意**。rulebook 参照でも可                         |

## Claude Code での起動例

```
@.claude/skills/verify-grill/SKILL.md @[要件ドキュメント]
[機能名] の検証設計だけ。実装はまだしない。
```

```
@.claude/agents/test-author.md @TEST_CONTRACT.md @[要件ドキュメント]
[機能名] の敵対的 E2E のみ。src/ は触らない。
```

```
@.claude/agents/spec-auditor.md @TEST_CONTRACT.md @[要件]
[機能名] を監査。TRACEABILITY を監査レポート本文として返す。保存は呼び出し側。FAIL なら完了拒否。
```
