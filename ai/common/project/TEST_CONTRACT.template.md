```text
rulebook 版: 2026-06-13
```

`TEST_CONTRACT.template.md` をコピーし、プロジェクト向けに調整する。  
実装者・テスト作者・verify-grill が従う契約（プロジェクトが用意する場合）。

**本ファイルが正本** — 否定テスト必須・S1–S8・実装側義務の詳細はここにのみ書く。  
`AGENTS.md` / `VERIFICATION.md` / スキルは要約 + 本ファイルへの参照に留める。

## 最優先原則

1. **テストは仕様の翻訳** — 成功パスの証明だけでは不十分
2. **否定テスト必須** — 機能 / マイルストーンごとに「できてはいけない」を最低 1 本（下記「共通レイヤー参照」可）
3. **ソルバー禁止** — 下記禁止パターンでクリアしない
4. **入力経路の一致** — 本番と同じ操作経路（タッチ / キーボード / API 等）
5. **完了の可視** — 内部状態 + **ユーザーに見える結果**を assert（可聴は下記「自動化不能」）

## 実装者のテスト容易性義務

テスト作者だけに義務を置いても S7 等は守れない。実装者は次を満たす。

- 乱数・盤面は**シード注入可能**にする（E2E で再現可能）
- 正解状態・制約は **data 属性等で観測可能**にする（テスト容易性。S1 条件付き許可の前提）
- アニメーションは E2E から完了を待てる（例: `transitionend`、テスト時短縮クエリ）
- `Math.random` 等の直接使用を避け、lint で検出可能にする
- 本番と同じイベント経路（Pointer / trusted touch 等）で操作可能な DOM 構造

## 禁止パターン（汎用）

| ID  | 禁止                                           | 理由                                     |
| --- | ---------------------------------------------- | ---------------------------------------- |
| S1  | 正解データを**無条件**にクリア手順へ使う       | 否定テストなしだとショートカットを見逃す |
| S2  | 1 操作あたり**許容以上の**状態ジャンプを見逃す | ワープ・一括更新の検出漏れ               |
| S3  | **アプリの正当な解集合を狭める**テスト側補正   | 実装バグを隠す                           |
| S4  | 本番と異なる入力 API のみで操作                | ヒットテスト・イベント経路の不一致       |
| S5  | 完了判定が内部状態のみで、**可視結果**を見ない | 演出・UX 欠落を見逃す                    |
| S6  | 失敗時の閾値緩和・skip・型/lint 無効化         | 品質ゲート回避                           |
| S7  | 固定 sleep で待機                              | フレーク・偽陽性。状態 / イベントで待つ  |
| S8  | 失敗後の禁止 UI / 副作用を未検証               | 「罰しない」「ロールバック」等の否定仕様 |

### S1 の条件付き許可

正解データ（シード・DOM・fixture）をソルバーで読むのは、**次の両方が同一スイートにある場合のみ**許可する。

1. **禁止操作の否定テスト**（違反方向へ操作しても状態不変）
2. **S2**（許容以上のジャンプを見逃さない — 下記補足）

共謀の定義は「データを読むか」ではなく **「実装が仕様違反でもこのテストは通るか」**。

### S2 補足（専用 assert 不要の場合）

S2 は専用の移動量 assert が無くても、**禁止方向への否定テスト**がワープ実装を検出すれば実質満たす。  
例: 「壁の方向へ進めない」— 到達可能セルへワープする実装でもキャラが動けば失敗する。

**注意**: 仕様上 1 操作で大きく進むのが正しい場合（例: あみだくじの 1 ドラッグ）は、「1 操作 1 ステップ」へ機械的に縮めない。**許容以上の**ジャンプが問題。

### S3 補足（仕様参照必須）

`rotateToZero(piece, 360)` のような形は、**回転対称がない（0° のみ正解）**仕様では正当。  
回転対称があるのに 0° 強制は S3 違反。grep だけでは区別できない — `sym` / 仕様を参照する。  
参考: `INCIDENT_CATALOG` の S3 ペア。

### S5 補足（可視 / 可聴）

| 種別                               | 契約                                                                 |
| ---------------------------------- | -------------------------------------------------------------------- |
| **可視**（紙吹雪・オーバーレイ等） | 自動 assert **必須**                                                 |
| **可聴**（効果音・ファンファーレ） | E2E 自動 assert は原則不可 → **自動化不能領域**に記載し実機/目視委譲 |

## 否定テスト必須（正本）

- 機能 / マイルストーンごとに否定を **最低 1 本**
- verify-grill で設計に落とす。実装前に記録
- **共通レイヤーで担保される場合** — 当該機能専用の否定が無くても、TRACEABILITY（監査成果物）で**参照テスト名**を明記すれば可（重複テストは避ける）

## 自動化不能領域

E2E で assert しない項目と委譲先を列挙する（未検証を構造化）。

| 項目                       | 委譲先                            |
| -------------------------- | --------------------------------- |
| 可聴（効果音・BGM）        | 実機確認 / 受け入れチェックリスト |
| 実機タッチ感               | 実機プレイ                        |
| [プロジェクトに応じて調整] |                                   |

## 差し戻し上限

監査 FAIL → test-author / implementer 差し戻しは、**同一不具合に 3 回**まで（プロジェクト憲法の停止条件と整合）。超えたらユーザーへ報告して停止。
