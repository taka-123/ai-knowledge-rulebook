# document-authoring

人が読む文章を、短く・分かりやすく・一貫したまま保つ。
アプリのコードそのものには使わない。
AI 向け指示（Skill / Agent / ルール / `CLAUDE.md`・`AGENTS.md`）の書き方は `ai-instruction-authoring` に任せる。文章としての短さ・一貫性は本 Skill も併用する。

## 目指すこと

- 読み手が次に何をすればよいか分かる
- 同じことを二度書かない
- 今回の会話だけの話を、恒久の本文に残さない
- 全体を直さず、末尾に注意だけ足して取り繕わない

## 会話残留の直し方

共有本文は、チャット文脈なしで意味が通ること。
指摘を受けて直すときは、弁明や補足を書き足すのではなく、問題の文を消して組み直す。

残りやすい型と例:

- 直前の指摘への反応 — 「なお、この構成でも問題ない」のような、会話相手に向けた弁明・言い直し
- 会話内だけの前提 — 「さっきの方針どおり」「今回の件では」など、その会話の外では何を指すか分からない参照
- 断定直後の弱め — 「必ず X する」の直後に、指摘のたびに継ぎ足した「ただし〜の場合は除く」が並ぶ

迷ったら「この文は、会話を知らない読み手に必要か」で判定する。唐突に見える文は残留の疑いが強い。

## 改良するときの見方

1. **短くする** — 重複や言い換えを削る
2. **中身は落とさない** — 字数は減っても、使う人が必要な情報は残す
3. **並べ直す** — 重要な順・依存関係の順に見出しを組み直す

## いつ全体を見直すか

小さな追記や誤字修正のたびに全文を読み直さない。次のときに全体を見る。

- 「見直して」「洗練して」「冗長を削って」と頼まれたとき
- 同じ文書を会話の中で複数回直したあと、一段落つけるとき
- 提出・共有・完了の直前

それ以外は、触った箇所の一貫性だけ確認する。

## 手順

1. 触る範囲を決める（全体見直しか、差分だけか）
2. 残す文は、消すと読み手の判断が変わるものだけ
3. 同じ決まりは一箇所にまとめる
4. 全体見直しのタイミングなら、会話残留・末尾パッチ・当たり前の念押しがないか確認する
5. 「見直して」→ 削る候補を理由付きで出す。「直して」→ 削るところまでやる

改良した本文はきれいな完成形で出し、何を変えたかの説明は会話側で行う。
