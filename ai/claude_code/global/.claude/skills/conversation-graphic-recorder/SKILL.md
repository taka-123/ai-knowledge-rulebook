# conversation-graphic-recorder

現在利用できる会話、資料、またはGit差分から、最終的に重要な内容を抽出し、
**1枚の日本語グラフィックレコーディング**として可視化する。

目的は会話ログや差分全文の保存ではない。
結論、関係性、流れ、比較、重要な条件、実際の変更点、次の行動を、
1枚で直感的に把握できるようにする。

## 重要な設計

このSkillはClaude Code、Codex、Cursorで同じディレクトリから読み込めるようにする。
ホスト製品や現在の環境で使える出力手段を判定し、次の順で処理する。

1. ネイティブ画像生成が利用可能なら、それを使ってPNGまたはWebPを生成する。
2. ネイティブ画像生成が利用できない場合は、編集可能な自己完結SVGを生成する。
3. ユーザーが「ネイティブ画像生成のみ」を指定し、それが利用できない場合は、
   完全な引き継ぎ用プロンプトをMarkdownファイルとして生成し、画像を生成したふりをしない。

したがって、Claude CodeでもこのSkillを読み込ませてよい。
Claude Codeでは通常SVGフォールバックが使われるが、外部画像生成ツールやSkillが接続されていればそれを優先できる。

## 基本方針

- 明示的に依頼されたときに使う。自発的に画像を生成しない。
- 特に指定がない限り、日本語で1枚だけ生成する。
- 入力にない事実、数値、因果関係、評価、結論、将来予測を追加しない。
- 会話内で訂正がある場合は、最後に確定した内容だけを採用する。
- 明確な結論がない場合は、無理に結論を作らず「暫定結論」と「未確定事項」を分ける。
- 画像内へ長文を入れない。
- 固有名詞、数字、日付、製品名、専門用語は可能な限り正確に保持する。
- 機密情報、認証情報、個人情報の値を画像や引き継ぎプロンプトへ載せない。
- 生成前に長い構成案を会話へ出さず、内部で整理して成果物を作る。

## 呼び出しと引数

引数がない場合は、依頼文と現在の文脈から対象、比率、スタイルを判定する。

### 対象

| 引数                   | 対象                                                                 |
| ---------------------- | -------------------------------------------------------------------- |
| `conversation`         | 現在のセッションで利用可能な会話                                     |
| `diff`                 | 現在のGit作業ツリー全体。staged、unstaged、確認可能なuntrackedを含む |
| `staged`               | staged差分だけ                                                       |
| `commit <rev>`         | 指定コミット                                                         |
| `range <base>..<head>` | 指定したコミット範囲                                                 |
| `file <path...>`       | 指定ファイルまたは複数ファイル                                       |

### 比率

| 引数                                | 比率        |
| ----------------------------------- | ----------- |
| 指定なし                            | 横長 `16:9` |
| `landscape` / `horizontal` / `16:9` | 横長 `16:9` |
| `portrait` / `vertical` / `4:5`     | 縦長 `4:5`  |

### スタイル

| 引数       | 用途                                         |
| ---------- | -------------------------------------------- |
| `general`  | 汎用                                         |
| `business` | 比較、採否、方針、ツール選定、変更要約       |
| `learning` | 学習、講義、概念理解                         |
| `roadmap`  | 手順、計画、移行、段階的変更                 |
| `system`   | 因果関係、構造、依存関係、アーキテクチャ変更 |
| `live`     | 会議、対談、議論                             |
| `story`    | 課題から解決・変化まで                       |
| `magazine` | スマートフォン、SNS、保存用縦長図解          |

### 出力バックエンド

| 引数     | 動作                                                                 |
| -------- | -------------------------------------------------------------------- |
| `auto`   | 既定。ネイティブ画像生成があれば使用し、なければSVG                  |
| `native` | ネイティブ画像生成だけを使用。なければ引き継ぎ用プロンプトを生成     |
| `svg`    | 画像生成モデルを使わず、編集可能なSVGを生成                          |
| `prompt` | 画像は生成せず、別ツールへ渡せる完全な生成プロンプトをMarkdownで保存 |

### その他

| 引数            | 動作                        |
| --------------- | --------------------------- |
| `output <path>` | 出力先を指定                |
| `no-open`       | 生成後に開かない            |
| `variants <n>`  | 複数案。明示時のみ。通常は1 |

`variants` は1〜3を基本とする。指定がなければ1枚だけ生成する。

例:

```text
/conversation-graphic-recorder conversation
/conversation-graphic-recorder diff landscape business
/conversation-graphic-recorder staged portrait magazine
/conversation-graphic-recorder range main..HEAD system
/conversation-graphic-recorder conversation svg
/conversation-graphic-recorder diff native output .work/artifacts/current-diff.png
```

ツールによって直接呼び出し記法が異なる場合も、同じ引数の意味で扱う。

## 使用する情報

優先順は次のとおり。

1. ユーザーが明示した対象、範囲、ファイル、Gitリビジョン
2. 現在のセッションで利用可能な会話
3. 現在の作業ディレクトリとGit情報
4. ユーザーが明示的に指定した添付資料
5. ユーザーが追加調査を明示した場合だけ外部情報

別セッションの会話を自動的に復元できると仮定しない。
対象が現在のコンテキストに存在しない場合は、存在しない内容を推測して生成しない。

## モード判定

次の語が主目的なら `diff` を選ぶ。

- 差分
- 変更点
- 何を変えたか
- 実際に変更されている内容
- working tree
- staged / unstaged
- commit / range

それ以外で、会話の結論、比較、方針、学習内容、調査結果の整理が主目的なら `conversation` を選ぶ。

## Procedure

### 1. 対象とオプションを確定する

- 対象モードを決める。
- 比率を決める。指定がなければ16:9。
- スタイルを決める。指定がなければ内容から自動選択する。
- バックエンドを決める。指定がなければ `auto`。
- 出力先を決める。

### 2. 根拠を収集する

#### conversation

現在の会話から次を抽出する。

- 最重要メッセージ
- 最終結論
- 重要ポイント3〜6個
- 採用、条件付き、不採用
- 比較、関係性、手順、時系列、課題と解決策
- 次に行うこと
- 再評価条件または未確定事項
- 必ず保持すべき固有名詞、日付、数値

除外するもの:

- 途中の試行錯誤
- 謝罪、相づち、言い直し
- 訂正前の情報
- 会話上の弁明
- 同じ意味の重複
- 最終判断に影響しない枝葉

#### diff

原則として次を確認する。

```bash
git status --short
git branch --show-current
git diff --stat
git diff
git diff --cached --stat
git diff --cached
```

untrackedファイルは、安全に読めるテキストだけ確認する。
巨大ファイル、バイナリ、生成物、依存物、機密らしいファイルは全文を読まない。

抽出する内容:

- 変更全体の目的を差分から確認できる範囲
- 実際に変わった振る舞い
- 主要な追加、変更、削除
- UI、API、DB、設定、依存関係、テスト、文書への影響
- 変更前から変更後への流れ
- staged / unstaged / untracked の状態
- 次に確認すべき点

ファイル名を大量に画像へ並べない。
主要なファイル名だけを残し、通常は機能・層・責務でまとめる。

品質の合否は判定しない。
差分から推測した影響は「推定」と明記する。

#### staged / commit / range / file

`conversation-visualizer` と同じ範囲解釈を使用する。

```bash
# staged
git diff --cached --stat
git diff --cached

# commit
git show --stat --summary <rev>
git show --format=fuller --find-renames <rev>

# range
git diff --stat <base>..<head>
git diff --find-renames <base>..<head>
git log --oneline --decorate <base>..<head>
```

### 3. Visual Briefを作る

画像生成またはSVG作成の前に、内部的に次の構造へ整理する。
会話へ長文で出力する必要はない。

```yaml
title: 内容を表す短いタイトル
core_message: 画像中央または最上部に置く最重要メッセージ
scope: 対象、範囲、確認時点
style: general | business | learning | roadmap | system | live | story | magazine
ratio: 16:9 | 4:5
blocks:
  - heading: 短い見出し
    points:
      - 短いフレーズ
      - 短いフレーズ
relations:
  - from: ブロック名
    to: ブロック名
    label: 関係を表す短語
next_action: 次の一歩
unresolved:
  - 未確定事項
must_preserve:
  - 固有名詞
  - 数字・日付
```

制約:

- 情報ブロックは3〜6個。
- 1ブロックの要点は原則1〜3個。
- 1つのテキスト要素を短くする。
- タイトルは1つ、最重要メッセージも1つ。
- 不明な内容を埋めない。

### 4. スタイルを選ぶ

指定がない場合:

- 比較、採否、方針、ツール選定、差分の変更要約: `business`
- 講義、書籍、学習、概念理解: `learning`
- 手順、成長、計画、段階的移行: `roadmap`
- 因果関係、構造、依存関係、ボトルネック: `system`
- 会議、対談、イベント、議論: `live`
- 課題から解決・変化まで: `story`
- スマートフォン閲覧、SNS、保存用縦長図解: `magazine`
- 判断できない場合: `general`

詳細は `references/style-guide.md` を読む。

### 5. バックエンドを選ぶ

#### auto

1. 現在のホストにネイティブ画像生成ツール、画像生成Skill、または明示的に接続された画像生成APIがあるか確認する。
2. 利用可能なら `native` と同じ処理を行う。
3. 利用できなければ `svg` と同じ処理を行う。

製品名や画像モデル名を固定しない。
現在使える機能を確認し、利用可能な画像生成手段を使う。

#### native

- 利用可能な画像生成ツールまたは画像生成Skillを使用する。
- Visual Briefと `references/style-guide.md` に基づく完全なプロンプトを渡す。
- 原則1枚だけ生成する。
- 指定比率を守る。
- 画像内の日本語は短くし、必ず保持すべき語を明示する。
- 写真風ではなく、手描き風のグラフィックレコーディングにする。
- 生成後、読み取れる範囲でタイトル、主要語、数字、日付の誤りを確認する。
- 明らかな誤字、欠落、レイアウト崩れがある場合は1回だけ再生成または修正を試みる。
- 画像生成ツールが出力場所を制御できない場合は、生成後に指定先へコピーまたは移動する。

ネイティブ画像生成が利用できない場合:

- SVGへ勝手に切り替えず、`prompt` と同じ引き継ぎ用ファイルを生成する。
- 「画像を生成した」とは報告しない。

#### svg

`assets/landscape-template.svg` または `assets/portrait-template.svg` を読み、
Visual Briefに合わせて要素を再配置し、自己完結SVGを作る。

- 横長: `1600 x 900`
- 縦長: `1200 x 1500`
- 外部画像、外部CSS、外部フォントへ依存しない。
- `<script>` を使用しない。
- システムフォントを使用する。
- 背景は白または淡い紙色。
- 色は3〜5色程度。
- 太めの線、角丸カード、矢印、囲み、簡単なアイコンを使う。
- 最小フォントサイズは原則24px。主要見出しは32px以上。
- テキストをカード外へはみ出させない。
- 1テキスト要素を最大2行程度にする。
- 文字量が多い場合は、文字を小さくするより情報を削る。
- 手描き感は、わずかな傾き、丸み、二重線などで表現してよいが、読みにくくしない。
- SVG内のテキストは選択・編集可能な `<text>` と `<tspan>` を基本とする。

SVGをPNGへ変換できるコマンドが既に環境にある場合は、ユーザーがPNGを求めたときだけ変換する。
新しい依存関係を勝手にインストールしない。

候補:

```bash
rsvg-convert input.svg -o output.png
magick input.svg output.png
```

#### prompt

画像生成用プロンプトをMarkdownファイルとして保存する。
ファイルには次を含める。

1. 対象と確認時点
2. Visual Brief
3. 画像生成ツールへそのまま渡せる完全な日本語プロンプト
4. 必ず保持する文字列
5. 禁止事項
6. 比率と推奨解像度
7. 想定出力ファイル名

秘密値や個人情報を含めない。

### 6. 出力ファイルを保存する

出力先の優先順:

1. `output <path>` またはユーザー指定パス
2. 現在のタスク用作業ディレクトリ内の `artifacts/`
3. リポジトリまたは作業ディレクトリ直下の `.work/artifacts/`

既定ファイル名:

```text
<内容を表すslug>-graphic-recorder.png
<内容を表すslug>-graphic-recorder.webp
<内容を表すslug>-graphic-recorder.svg
<内容を表すslug>-graphic-recorder-prompt.md
```

ファイル名は英小文字のkebab-caseを基本とする。
生成物を勝手にGit管理へ追加しない。
`.gitignore` も勝手に変更しない。

### 7. 検証する

#### 共通

- ファイルが存在する。
- 指定比率になっている。
- タイトルが1つある。
- 最重要メッセージが1つある。
- 情報ブロックが3〜6個に収まっている。
- 入力にない事実が増えていない。
- 固有名詞、数字、日付が可能な範囲で正しい。
- 秘密値や個人情報が露出していない。

#### native

- 主要な日本語が読める。
- 小さい文字を大量に配置していない。
- 写真風や3D風へ逸脱していない。
- 同じ内容を複数箇所へ重複していない。

#### svg

同梱スクリプトを使用できる場合:

```bash
python3 scripts/validate_svg.py <生成したSVG>
```

実行位置がSkillディレクトリ外の場合は、Skill内のスクリプトへ絶対パスまたは正しい相対パスを指定する。

### 8. 提示する

- 生成したファイルパスを示す。
- macOSで `no-open` がなく、コマンド実行が許可される場合は `open <path>` を実行する。
- 実行しない場合は、次のコマンドを短く示す。

```bash
open <path>
```

ネイティブ画像を生成できず引き継ぎ用プロンプトだけを作った場合は、その事実を明記する。
画像を生成したふりをしない。

## 共通デザイン

- 1枚絵
- 手描き風のグラフィックレコーディング
- 白または淡い紙色の背景
- 太めで読みやすい線
- 色は3〜5色程度
- 子ども向けになりすぎない
- 装飾より情報理解を優先する
- タイトルを上部に大きく配置する
- 中央に最重要メッセージまたは全体像を置く
- 周囲に3〜6個の情報ブロックを配置する
- 矢印、囲み、吹き出し、番号、簡単な人物・道具・概念アイコンを使う
- 視線が自然に流れるレイアウトにする
- 最後に「まとめ」「次のアクション」「次の一歩」のいずれかを配置する

## 画像内テキスト

- 日本語を基本とする。
- 長文を入れず、短い見出しと短いフレーズへ圧縮する。
- 文字は大きく、読みやすくする。
- 情報を詰め込みすぎない。
- 画像内テキストは必要最小限にする。
- 固有名詞、数字、日付、製品名、専門用語は可能な限り正確に保持する。
- 誤字を減らすため、1つのテキスト要素を短くする。
- 英語が正式名称の場合は英語を保持し、必要に応じて短い日本語説明を添える。
- 入力にないキャッチコピーを作らない。

## 禁止事項

- 入力にない情報を補完しない。
- 長文を画像内へそのまま入れない。
- 小さい文字を大量に配置しない。
- 写真風、3D風、リアル人物写真風にしない。
- 装飾を増やして読みづらくしない。
- 意味のないチャートや数値グラフを作らない。
- 出典URLを画像内に小さく詰め込まない。
- 同じ内容を複数箇所に重複させない。
- ユーザー名や会話上の経緯を不要に含めない。
- ホスト製品や画像モデル名を固定し、利用可能性を確認せず呼び出さない。
- ネイティブ画像生成がない環境で、画像生成済みと偽らない。

## 他Skillとの境界

| Skill                           | 責務                                                 |
| ------------------------------- | ---------------------------------------------------- |
| `conversation-graphic-recorder` | 確定内容または実変更点を1枚の画像・SVGで直感的に示す |
| `conversation-visualizer`       | 同じ情報を詳細な単一HTMLで示す                       |
| `diff-quality-gate`             | 差分の品質、残骸、機密、明らかな欠陥を評価する       |
| `document-authoring`            | 正本となる自然言語文書を作成・編集する               |
