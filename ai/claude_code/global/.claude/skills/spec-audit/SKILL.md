# spec-audit

## いつ使うか

- 検証（自動テスト・再現確認等）が完了した**後**
- 完了報告・PR 提出の**前**
- **実装者の会話文脈を引き継がない**別セッション / サブエージェントとして

前提: 対象プロジェクトに `TEST_CONTRACT.md` と `TRACEABILITY.template.md` が存在すること。無い場合は本スキルを適用しない。

## 手順

### 1. TRACEABILITY を生成（常設表の更新ではない）

`TRACEABILITY.template.md` に従い、**この監査の成果物**として対応表を新規作成する。

- **監査レポート本文**として TRACEABILITY を返す（`docs/audits/` への保存は呼び出し側）
- 保存先の例: `docs/audits/TRACEABILITY-M[N].md`
- 各行: 要件引用、実装根拠、肯定検証、否定検証（または共通レイヤー参照）
- 否定列が空で参照も無効 → **完了拒否**

### 2. TEST_CONTRACT 監査

`TEST_CONTRACT.md` の S1–S8・実装容易性・自動化不能欄を確認。

**grep だけに頼らない。** 不変条件ベースで判断する。

| 確認               | 方法                                       | 誤検出例 / 正当例                  |
| ------------------ | ------------------------------------------ | ---------------------------------- |
| ショートカット実装 | 否定テスト + S2 実質担保                   | 公開用 data 属性名の一致だけ       |
| 解集合の狭め (S3)  | 仕様が許す解の範囲を参照                   | 下記ペア                           |
| 否定テスト         | 否定列または共通レイヤー参照               | —                                  |
| 可視完了 (S5)      | 可視 assert があるか                       | 内部状態のみ。可聴は自動化不能で可 |
| 各テスト           | 「実装が壊れていたら失敗する条件」が妥当か | —                                  |

**S3 ペア（判定例。プロジェクト固有の実例は `INCIDENT_CATALOG` 参照）**

| 例                                                                                                            | 判定        |
| ------------------------------------------------------------------------------------------------------------- | ----------- |
| 仕様が複数の正解を許すのに、テストが期待する特定解へ実装を固定（例: 順序未規定の API をテストの期待順に固定） | S3 **違反** |
| 仕様自体が解を一意に定めている場合の固定（例: 仕様で昇順と明記済み）                                          | S3 **正当** |

### 3. 上位品質チェック

DoD と上位完成度（SPEC 等にあれば）のギャップを短文で報告。

```
## 監査結果: [PASS / FAIL]

### 満たした要件行
-

### ギャップ（FAIL 時）
-

### 推奨アクション
- test-author:
- implementer:
```

### 4. 監査結果が FAIL のとき

- 「マイルストーン / 機能 未完了」と伝える
- ユーザーへの完了報告を**止める**
- 同一不具合の差し戻しが `TEST_CONTRACT.md` の上限（通常 3 回）を超えたらユーザーへ報告

## 禁止

- 「テストが通ったから PASS」だけの判定
- ギャップを曖昧にして PASS とすること
