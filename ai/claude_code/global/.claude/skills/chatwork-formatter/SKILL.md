# chatwork-formatter

Markdown を ChatWork 記法に変換し、コピー可能ブロックとして会話内に提示する。
デフォルトはプレーンテキスト。`[info]` は枠で囲むことが読みやすさに寄与するドキュメント系に限って使う。

## Procedure

1. `docs/transform-rules.md` を Read で参照する（記法の対応表）。
2. メッセージ種別を判定する（下記「`[info]` の判定」）。
3. 対応表に従って記法を置き換える。「変換で形式を増幅しない」を守る。
4. 「👇 CWにそのままコピーしてください」を 1 行出力し、続けて変換済みテキストをコードブロックで提示する。

## `[info]` の判定

見出しやリストの有無では判定しない。メッセージ種別で判定する。

| 種別           | 典型例                                                     | 記法                                                            |
| -------------- | ---------------------------------------------------------- | --------------------------------------------------------------- |
| 会話系         | 返信、依頼、確認、質問、お礼、短い報告                     | プレーン。`[info]` は使わない。見出しがあれば `【】` `▼` で表現 |
| ドキュメント系 | 手順書、共有資料、複数セクションの案内、長い構造化レポート | 全体を `[info]` で囲む。冒頭の H1 は `[title]` に変換           |

種別が判然としない場合の補助基準:

- その場で読まれて流れていく文 → プレーン。
- 後から参照・保存・実行される文 → `[info]`。
- それでも迷ったら `[info]` を使わない（デフォルトはプレーン）。

## 変換で形式を増幅しない

本 Skill の責務は記法の置き換えのみ。元文にない形式を足すと不自然な「AI くさい」出力になる。

- 元文の意味・構成・項目数を変えない。平文は平文のまま、2 項目を 3 項目に増やさない。
- 変換で新たに見出し・箇条書き・セクション・ラベル列挙を作らない。
- 会話系のメッセージに `▼` 見出しや記号装飾を追加しない。
- 絵文字はユーザーから指示された場合のみ追加する。

## ルール（必守）

- ファイルへ保存しない（会話内提示のみ）。
- `[title]` は `[info]` の内側でのみ使う。
- ChatWork で表示できない記法は削除・平文化する（崩れたまま渡さない）。

## Examples

### Example 1: 日常の返信（会話系 → `[info]` なし）

Input:

```markdown
ご確認ありがとうございます。**明日の午前中** に修正して、完了したらご連絡します。
```

Output:

👇 CWにそのままコピーしてください

```
ご確認ありがとうございます。明日の午前中に修正して、完了したらご連絡します。
```

判定: 返信（会話系）。太字の記号を除去するだけで、構造は足さない。

### Example 2: 短い依頼（会話系 → リストがあっても `[info]` なし）

Input:

```markdown
お疲れさまです。リリース前に以下をお願いできますか。

- ステージングでの動作確認
- 確認結果のこのスレッドへの返信
```

Output:

👇 CWにそのままコピーしてください

```
お疲れさまです。リリース前に以下をお願いできますか。

・ステージングでの動作確認
・確認結果のこのスレッドへの返信
```

判定: 依頼（会話系）。箇条書きは `・` へ変換するが、リストの存在は `[info]` を使う理由にならない。

### Example 3: 構造化レポート（ドキュメント系 → `[info]` あり）

Input:

```markdown
# 4月度 定例報告

## 進捗

- 設計完了
- 実装 70%

## 課題

- API 仕様の確定待ち

## 来月の予定

- 結合テスト開始
```

Output:

👇 CWにそのままコピーしてください

```
[info]
[title]4月度 定例報告[/title]

▼ 進捗
・設計完了
・実装 70%

▼ 課題
・API 仕様の確定待ち

▼ 来月の予定
・結合テスト開始
[/info]
```

判定: 複数セクションを持ち、後から参照される報告書（ドキュメント系）。枠で囲むことが読みやすさに寄与する。

### Example 4: 手順書 + コードブロック（ドキュメント系 → `[info]` + `[title]`）

Input: H1 見出し + 番号付きリスト + bash コードブロック 2 つを含む Markdown。

Output:

👇 CWにそのままコピーしてください

```
[info]
[title]セットアップ手順[/title]

1. 依存をインストール
[code]npm install[/code]

2. 起動
[code]npm run dev[/code]
[/info]
```

判定: 手順書（ドキュメント系）。実行しながら参照されるため `[info]` で囲み、H1 を `[title]` に変換する。
