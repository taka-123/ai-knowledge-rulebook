# conversation-visualizer

現在利用できる会話、資料、またはGit差分から、確定内容と実変更点を抽出し、
**ブラウザで単独閲覧できる1つの自己完結HTML**へ変換する。

「読み物かレビュー資料か」でSkillを分けず、情報の型に応じてHTML内の構成を変える。

## 基本方針

- 明示的に依頼されたときに使う。HTMLを自発生成しない。
- 会話ログや差分全文をそのまま転記しない。
- 最終的に確定した内容、実際に変更された内容、判断に必要な根拠を残す。
- 元の会話やリポジトリを知らない人でも、HTML単体で理解できるように組み直す。
- 事実、数値、日付、固有名詞、ファイル名を勝手に変更しない。
- 入力にない結論、効果、因果関係、将来予測を追加しない。
- 品質評価を求められていない場合、変更の良し悪しを判定しない。
- 機密情報、認証情報、個人情報の値はHTMLへ載せない。

## 呼び出しと引数

引数がない場合は、依頼文と現在の文脈から対象を判定する。
判定できない場合は、現在の会話を対象とする。

| 引数                   | 対象                                                                 |
| ---------------------- | -------------------------------------------------------------------- |
| `conversation`         | 現在のセッションで利用可能な会話                                     |
| `diff`                 | 現在のGit作業ツリー全体。staged、unstaged、確認可能なuntrackedを含む |
| `staged`               | staged差分だけ                                                       |
| `commit <rev>`         | 指定コミット                                                         |
| `range <base>..<head>` | 指定したコミット範囲                                                 |
| `file <path...>`       | 指定ファイルまたは複数ファイル                                       |
| `output <path>`        | 出力先を指定                                                         |
| `no-open`              | 生成後にブラウザを開かない                                           |

例:

```text
/conversation-visualizer conversation
/conversation-visualizer diff
/conversation-visualizer staged output .work/artifacts/staged-changes.html
/conversation-visualizer range main..HEAD
/conversation-visualizer file docs/decision.md
```

ツールによって直接呼び出し記法が異なる場合も、同じ引数の意味で扱う。

## 使用する情報

優先順は次のとおり。

1. ユーザーが明示した対象、範囲、ファイル、Gitリビジョン
2. 現在のセッションで利用可能な会話
3. 現在の作業ディレクトリとGit情報
4. ユーザーが明示的に指定した添付資料
5. ユーザーが追加調査を明示した場合だけ外部情報

別セッションの会話を自動的に復元できると仮定しない。
対象が現在のコンテキストに存在しない場合は、その事実を短く伝え、利用可能なログまたは資料を求める。

## モード判定

次の語が主目的なら `diff` を選ぶ。

- 差分
- 変更点
- 何を変えたか
- 実際に変更されている内容
- working tree
- staged / unstaged
- commit / range

それ以外で、会話の結論、比較、方針、学習内容、調査結果の整理が主目的なら `conversation` を選ぶ。

## Procedure

### 1. 対象を確定する

- 引数と依頼文からモードを確定する。
- 出力先の指定を確認する。
- 対象がない場合は、存在しない内容を推測して作らない。

### 2. 根拠を収集する

#### conversation

現在の会話から次を抽出する。

- 最終結論
- 採用した案、条件付き採用、不採用
- 判断理由
- 比較軸と比較結果
- 決定済みの手順、役割分担、運用
- 次に行うこと
- 未確定事項、再評価条件
- 必ず保持すべき固有名詞、数値、日付、対象プラン、環境

除外するもの:

- 訂正前の情報
- 途中で撤回された案
- 試行錯誤の経緯だけの記述
- 謝罪、相づち、会話上の弁明
- 同じ意味の重複
- 最終判断に影響しない枝葉

明確な結論がない場合は、結論を作らず「暫定整理」と「未確定事項」を分ける。

#### diff

原則として次を確認する。

```bash
git status --short
git branch --show-current
git diff --stat
git diff
git diff --cached --stat
git diff --cached
```

untrackedファイルは、一覧を確認したうえで、テキストとして安全に読めるものだけ内容を確認する。
巨大ファイル、バイナリ、生成物、依存物、機密らしいファイルは全文を読まない。

抽出する内容:

- 変更の目的を差分から確認できる範囲
- 実際に変わった振る舞い
- 追加、変更、削除された主要機能
- ファイル単位ではなく、機能・層・責務でまとめた変更群
- UI、API、DB、設定、依存関係、テスト、文書への影響
- staged / unstaged / untracked の区別
- 移行、互換性、運用上の注意
- 差分だけでは断定できない点

次はしない。

- テストやlintの実行。ただしユーザーが同時に依頼した場合は除く。
- 差分品質の合否判定。必要なら `diff-quality-gate` を併用する。
- 変更理由をコードから推測して事実扱いする。
- 秘密値の転載。

差分から推測した影響は「推定」と明記する。

#### staged

```bash
git diff --cached --stat
git diff --cached
```

#### commit

```bash
git show --stat --summary <rev>
git show --format=fuller --find-renames <rev>
```

#### range

```bash
git diff --stat <base>..<head>
git diff --find-renames <base>..<head>
```

必要に応じてログも確認する。

```bash
git log --oneline --decorate <base>..<head>
```

#### file

指定されたファイルだけを読み、会話に依存せず理解できる形へ再構成する。
複数ファイルの内容が矛盾する場合は、矛盾を隠さず「未確定」または「要確認」とする。

### 3. 情報を選別する

残す情報の優先順:

1. 内容を表すタイトル
2. 最重要メッセージ
3. 最終結論または変更の要約
4. 決定事項または主要変更点
5. 比較、構造、フロー、時系列
6. 影響範囲と注意点
7. 次に行うこと
8. 未確定事項、再評価条件
9. 根拠、対象範囲、確認時点

すべてのセクションを埋める必要はない。
内容がないセクションは削除する。

### 4. HTMLを生成する

必ず `assets/template.html` を読み、デザインとコンポーネントを再利用する。
`<style>` は無変更で全文コピーし、`.shell` 内だけを組み替える。色はトークン（ok / warn / danger / accent）経由のみ使用し、ライト固定とする。
テンプレートは固定レイアウトではない。必要なセクションだけ残し、順序も内容に合わせて組み替える。

#### 共通ルール

- 単一の自己完結HTMLにする。
- 外部CSS、外部JavaScript、外部フォント、CDN、外部画像へ依存しない。
- 原則としてJavaScriptを使わない。
- `<html lang="ja">` とUTF-8を使用する。
- `<title>` は内容を表す具体的なものにする。
- レスポンシブ表示と印刷表示を壊さない。
- 見出しは日本語を基本とする。正式名称とコード識別子は原文を保持する。
- プレースホルダーを残さない。
- 空のカード、空の表、空のセクションを残さない。
- 長い詳細、ファイル一覧、根拠は `details` に畳んでよい。
- 最初の画面で「何について」「結論または変更要約」「重要点」が分かるようにする。
- HTML全文をチャットへ貼らず、ファイルとして保存する。

#### 情報型とコンポーネント

| 情報の型                       | テンプレート上の表現                                        |
| ------------------------------ | ----------------------------------------------------------- |
| タイトル、種別、対象、確認時点 | `.hero` / `.kicker` / `.doc-meta`                           |
| 結論、変更要約                 | `.verdict`（状態で ok / warn / danger）                     |
| 判断に効く数値・条件           | `.facts`（あるときだけ。埋め草を作らない）                  |
| 収録内容の一覧                 | `.contents`（セクションが複数あるとき任意）                 |
| 採用・不採用、使う・使わない   | `.split` + `.panel`（ok・✓ / warn・✕）                      |
| 案の対置、役割分担             | `.cols` + `.card`（推奨案は `.card.recommended`）           |
| 状態・判定・重大度             | `.badge` ＋ 行に色付けした `table`（tr.ok / warn / danger） |
| 比較                           | 比較 `table`（`.table-wrap` で包む。比較軸を行に）          |
| 手順、処理フロー               | `.flow`（分岐は `.flow-branch` で並列に描く）               |
| 時系列、変更経過               | `.timeline`（重大イベントは li に色クラス）                 |
| 注意、未確定、制約             | `.callout`（warn / danger）                                 |
| 詳細、ファイル一覧、根拠       | `details`（既定は閉じる）                                   |

#### conversation向けの標準構成

必要なものだけ使う。

1. タイトルと対象
2. 最終結論
3. 決まったこと
4. 比較または判断理由
5. 採用する流れ・運用
6. 次に行うこと
7. 未確定事項
8. 前提・根拠

#### diff向けの標準構成

必要なものだけ使う。

1. タイトル、対象ブランチ、状態
2. 変更全体の要約
3. 実際に変わったこと
4. 変更領域別の詳細
5. 影響範囲
6. staged / unstaged / untracked の状態
7. 移行・確認が必要な点
8. 主要ファイル一覧と根拠

ファイル名の羅列を本文の中心にしない。
「どのファイルが変わったか」より先に「何が変わったか」を示す。

### 5. 保存する

出力先の優先順:

1. `output <path>` またはユーザー指定パス
2. 現在のタスク用作業ディレクトリ内の `artifacts/`
3. リポジトリまたは作業ディレクトリ直下の `.work/artifacts/`

ファイル名は内容を表す英小文字のkebab-caseを基本とする。

例:

```text
.work/artifacts/auth-flow-decision.html
.work/artifacts/current-diff-summary.html
.work/artifacts/release-range-summary.html
```

生成物は原則として使い捨てであり、勝手にGit管理へ追加しない。
`.gitignore` も勝手に変更しない。

### 6. 検証する

保存後に次を確認する。

- プレースホルダーが残っていない
- HTMLの開始・終了タグが対応している
- 空セクションがない
- 最初の画面で要点が分かる
- 元の情報にない事実が増えていない
- 秘密値や個人情報が露出していない
- ファイルが実際に存在する

同梱スクリプトを使用できる場合:

```bash
python3 scripts/validate_html.py <生成したHTML>
```

実行位置がSkillディレクトリ外の場合は、Skill内のスクリプトへ絶対パスまたは正しい相対パスを指定する。

### 7. 提示する

- 生成したファイルパスを示す。
- macOSで `no-open` がなく、コマンド実行が許可される場合は `open <path>` を実行する。
- 実行しない場合は、次のコマンドを短く示す。

```bash
open <path>
```

説明は、対象、生成先、重要な制約があればそれだけにする。
HTML全文を会話へ再掲しない。

## 他Skillとの境界

| Skill                           | 責務                                           |
| ------------------------------- | ---------------------------------------------- |
| `conversation-visualizer`       | 確定内容または実変更点をHTMLで理解しやすくする |
| `conversation-graphic-recorder` | 同じ情報を1枚の画像またはSVGで直感的に示す     |
| `diff-quality-gate`             | 差分の品質、残骸、機密、明らかな欠陥を評価する |
| `diff-review-commit`            | レビュー後にコミットまで行う                   |
| `document-authoring`            | 正本となる自然言語文書を作成・編集する         |

## 禁止事項

- HTMLを自発生成する長いゲート判定を持ち込まない。
- 会話や差分にない内容を補完しない。
- 変更点の説明と品質判定を混同しない。
- 生成物を正本Markdownの代わりにしない。
- 既存コードや文書を、このHTML生成のためだけに変更しない。
- ユーザーが求めていない外部調査を行わない。
