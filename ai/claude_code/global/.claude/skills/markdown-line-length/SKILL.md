# markdown-line-length

Markdown の行長を適切に保ち、
ソースでもレンダリング後でも読みやすくする。

## 適用タイミング

- `.md` / `.mdx` / `.mdc` の作成・編集時に常に適用
- 既存 Markdown の行が長く読みにくいと指摘されたとき

## 適用しないとき

- Markdown 以外の編集、または読み取り専用のとき

## 核心ルール

### 行長の目安

- 目安は 80〜120 文字。厳密な上限ではなく読みやすさが基準。
- URL・テーブル・コードブロック・見出しは例外として超過可。

### 平文（段落テキスト）

句点（。）やピリオド（.）の後、
または意味の切れ目で改行してよい。
Markdown では単一改行はスペース扱いになり、
レンダリングに影響しないため、
ソースの可読性のために自由に使える。

### 箇条書き（リスト項目）— 最重要

**箇条書きの途中で物理改行を入れてはならない。**

Markdown のリスト項目では、途中に改行を入れても
レンダリング時は 1 行に結合される。
つまり見た目は何も変わらないのに、
ソースだけが不自然に分断される。
これが最も避けるべきアンチパターンである。

長い箇条書きへの対処法（優先順）:

1. **ネストで分割** — 情報を子項目に分けて構造化する
2. **文を短く書き直す** — 冗長な表現を削り 1 行に収める
3. **そのまま残す** — 120 文字程度なら無理に短縮しなくてよい

NG — 箇条書き途中の物理改行（レンダリングでは 1 行）:

```markdown
- 既存のAIレビューエージェント
  （review-core-diff-gatekeeper等）を
  GitHub Actions CI に組み込み、PR作成時に自動で
  一次レビューを実行する。
```

OK — ネストで構造化:

```markdown
- AIレビュー自動化のCI統合
  - review-core-diff-gatekeeper等をGitHub Actionsに組み込む
  - PR作成時に一次レビューを自動実行
```

OK — 文を短縮して 1 行に:

```markdown
- 既存AIレビューエージェントをGitHub Actions CIに統合し、PRの一次レビューを自動化する。
```

### 見出し

短ければ 1 行のまま。極端に長い場合のみ短くする。

## まとめ

| 要素      | 方針                                     |
| --------- | ---------------------------------------- |
| 平文      | 句読点・意味の切れ目で改行可             |
| 箇条書き  | **途中改行は禁止**。ネスト or 短縮で対処 |
| URL・表等 | 行長制限の対象外                         |
| 見出し    | 基本 1 行。長すぎれば短くする            |
