# html-web-report

長い Markdown を、ブラウザで要点から掴める HTML にする。
価値は **スクロールなしの最初の画面で「何か・いつ・何が分かるか」が完結すること**。
見た目は `html-review-artifact` と共通テンプレ。用途は通読・共有の要点把握。
元文の意味・事実・数値は変えない。正本の置き換えもしない。

判断・比較・インシデント資料は `html-review-artifact` に任せる。

## Procedure

1. **経路判定**:
   - 除外（即 No）に該当 → HTML を作らず代替を提示して終了。
   - **明示依頼あり** → 除外に該当しなければ生成。
   - **明示依頼なし** → 除外後に Gate criteria で判定。Yes なら生成（提案で止めない）。No なら Markdown のまま。
   - 主目的が意思決定・複数案比較・インシデント判断なら `html-review-artifact` を使う。
2. **正本の確認**: 正本 MD がある → HTML は生成物（正本は編集しない）。ない → 使い捨てなら HTML のみ可。継続更新文書なら正本 MD の新設を先に提案する。
3. **生成**: 単一 self-contained HTML。`assets/template.html` を読み、`<style>` は無変更で全文コピーし、`.shell` 内の body だけを組み替える。ライト固定（テーマ切替・JS なし）。
4. **配置と提示**: 「Output destination」に保存し `open <path>` を示す。会話内への HTML 全文出力は行わない。

## Gate criteria（明示依頼がない場合）

デフォルトは **No**。

| 除外（即 No）              | 典型例                               | 代替                                      |
| -------------------------- | ------------------------------------ | ----------------------------------------- |
| 意思決定・比較レビューが主 | 2 案比較、インシデント判断、推奨決定 | html-review-artifact                      |
| 正本の作成・置き換えが目的 | ADR・実装計画を HTML で書く          | 正本は Markdown。理解用ビューは別途判定   |
| 機械処理が主目的           | テスト一覧、API 契約、設定スキーマ   | JSON / JSON Schema                        |
| 交流・会話系               | 返信、依頼、確認、短い CW            | chatwork-formatter またはプレーンテキスト |
| 短文で MD で足りる         | 数行の進捗、短いメモ                 | Markdown のまま                           |

除外に当てはまらない場合、次の **即 Yes** のいずれかなら生成する。

| 即 Yes                                       | 理由                       |
| -------------------------------------------- | -------------------------- |
| 明示的に読む・配布する長文レポートが主目的   | ブラウザ閲覧向き           |
| 手順書・ニュースレター・まとめ記事として共有 | 段落・見出しの読み物が本体 |

即 Yes でない場合、次の 3 問のうち **2 つ以上 Yes** なら生成する。

| 問い                                                  | Yes なら  |
| ----------------------------------------------------- | --------- |
| 見出しが複数あり、スクロールして通読する長さか        | HTML 候補 |
| 非エンジニアや後から見返す人向けの共有か              | HTML 候補 |
| 表・補足・状態・進捗があり、MD のままだと追いにくいか | HTML 候補 |

迷ったら **No**。

## Content boundary

- してよい: 最初の画面での要点整理、見出しの日本語化、提示順の組み替え、詳細の折りたたみ、表の横スクロール化、本文にある状態・進捗の表示。
- してはいけない: 元文にない事実・数値の追加、意味を変える要約、Markdown の全セクション転記、詳細を最初から全部開くこと。

## HTML generation rules

目的: **最初の画面で用が足りる**こと。全文転記の置き換えではない。

1. `assets/template.html` の `<style>` は無変更。色はトークン経由のみ。ライト固定。
2. **ヒーロー（必須）** — `.hero` に、種別 `.kicker`・`h1`・`.doc-meta`・`.verdict`・`.facts` をまとめる。この 1 ユニットで「何か／結論／要点」が分かること。無い情報は作らない。
   - `.verdict` … 一文の結論（色付き面＋左罫。状態で ok / warn / danger）
   - `.facts` … 何か／いつ／期待（`.fact` 3 枚を基本。ヒーロー下部に埋め込み）
   - `.contents` … 任意。大見出しが複数あるとき、何が載っているかの誤解を防ぐ 1 行ストリップ
3. **`.split` + `.panel`** … 使い分けが明確なとき、使うとき（ok・✓）／使わないとき（warn・✕）をヒーロー直下に置く。
4. **詳細は `details`（閉じた状態）** — 手順全文、長い表、一覧。ユーザーが「詳細も全部開いて」と言ったときだけ最初から開く。
5. 目次は原則付けない。見出しは日本語（固有名詞・コード名は除く。`h2` は自動採番される）。
6. 部品:

| 情報の型                               | コンポーネント                              |
| -------------------------------------- | ------------------------------------------- |
| ヒーロー（種別・結論・要点）           | `.hero` / `.kicker` / `.verdict` / `.facts` |
| 収録内容（任意）                       | `.contents`                                 |
| 使う／使わない（使い分けが明確なとき） | `.split` + `.panel`（ok / warn）            |
| 注意・警告                             | `.callout.warn` / `.callout.danger`         |
| 詳細・長い一覧                         | `details`（既定は閉じる）                   |
| 表                                     | `.table-wrap`                               |
| 状態（本文にあるもの）                 | `.badge`                                    |

7. 提出前: スクロールせずに「何か／結論／使いどころ」が言えるか。色付き面に置いた `code` が読めるか。詳細が開きっぱなしでないか。元文にない事実が増えていないか。

## Output destination

1. ユーザー指定パス（`docs/` 等の正本ディレクトリは避ける）
2. 進行中タスクの作業ディレクトリ直下
3. `.work/artifacts/<スラッグ>.html`

保存後、`open <path>` を提示。生成物は原則 `.gitignore` 対象。
