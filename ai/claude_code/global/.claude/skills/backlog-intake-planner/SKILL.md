# backlog-intake-planner

薄い・曖昧な要求を、起票して意図を確認できる `requirements.md` と、そのまま着手できる `implementation-plan.md` に起こす。
要求の明確化から要件定義・実装計画までを、軽くも深くも、段階的に進める。

## 生成物

- `requirements.md` — 前半は確認用サマリー（非エンジニア向け・専門用語なし。意図の合致と完成イメージを確認できる）、後半は実装・レビュー用（要件詳細・受け入れ条件・DB変更・レビュー観点。技術用語可）。Backlog 記法で起票に貼れる形。
- `implementation-plan.md` — requirements.md と対で読めば、別資料なしで上から実行して完成できる実装計画／スプリント計画（着手ブロッカー・データモデル・契約・制御ルール／制約・実装ステップ・完了確認）。深掘り時のみ生成する。

雛形は `docs/template-guide.md` を参照する。

## モード

- **軽量（既定）**: `requirements.md` の前半（概要・完成イメージ・やらないこと・前提・確認したいこと）を最大5問・平易な質問で埋め、Draft 起票する。
- **深掘り**: `requirements.md` の後半（要件・受け入れ条件・DB変更・レビュー観点）と `implementation-plan.md` を、技術質問も使って埋める。要件は「要件1 / 要件2」と番号を振り、受け入れ条件から対応付ける。
- **モードの選び方**: 既定は軽量。ユーザーが起動時に「設計まで」「実装計画も」「深掘りして」等と言えば最初から深掘り。指定がなければ軽量で進め、**軽量の起票が終わった最後に深掘り要否を必ず尋ねる**（進め方 9）。勝手に深掘りを始めない／勝手に省略しない。

## 深掘りと grill-me

深掘り（技術・仕様の詰め）は次のどちらか。軽量モード・Backlog 起票 SOP（配置・ブランチ・確認依頼文）は**どちらでも本スキルが担う**。

| 経路                 | いつ                                           | 壁打ち                                                        | テンプレ反映                                                                           |
| -------------------- | ---------------------------------------------- | ------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **A. 本スキル内**    | 深掘り指定で grill-me 未実施                   | 進め方 5 の質問                                               | そのまま生成                                                                           |
| **B. grill-me 連携** | ユーザーが grill-me 済み／「もう詰めた」と明示 | **grill-me** に任せる（本スキルでは深掘り質問を繰り返さない） | grill-me のまとめ + `docs/template-guide.md` で後半・`implementation-plan.md` を埋める |

経路 B の手順:

1. grill-me の「決まったこと / 前提・制約 / 未決」を読み、推測は「前提」、未決は「確認したいこと」へ（確定事項と混ぜない）。
2. `docs/template-guide.md` に沿って `requirements.md` 後半と `implementation-plan.md` を生成する。
3. Backlog 記法の最終調整が必要なら **backlog-markdown-formatting** を適用する（意味は変えない）。
4. 実装着手前の検証設計は **verify-grill**（grill-me の「次」と同型）。

経路 B でも `.work/backlog/BUSSW-AI/` 配置・ブランチ案・起票後手順・確認依頼文は本スキルどおり案内する。

## 必須情報

起票には最低限「目的・背景」と「やりたいこと（機能・修正の内容）」が要る。揃うまで壁打ちで補い、それでも埋まらない分は推測で補って「前提」欄に明記する。

## 進め方

1. 要求文（読めるならリポジトリも）から、推測で埋められる前提を埋め、`requirements.md` の「前提（こちらで補った・要確認）」欄に**推測として明記**する。要求者の確定回答とは分ける。
2. 残る不明点のうち、仕様の核に効くものを選ぶ（この機能で合っているか／この機能は不要では／画面・UI はどうあるべきか）。
3. 軽量モードは、その中から**最大5問**を選択式で1回にまとめて聞く。平易な言葉で、コード・DB・技術には触れない。
4. 5問で埋まらなくても、またスキップされた質問があっても **Draft 起票して終える**。残り・スキップ分は「確認したいこと / 未確定」へ**宛先付き**で置く（後でまとめて聞ける形にする）。
5. 深掘りはユーザーが明示したときだけ（**grill-me 済みなら「深掘りと grill-me」経路 B**。未実施なら経路 A）。`requirements.md` 後半と `implementation-plan.md` を埋める。`implementation-plan.md` は §2〜4（データモデル・契約・制御ルール／制約）を確定してから §5 実装ステップを書き、ステップからそれらを参照させる。
6. ブランチ名を提案する（下記）。
7. 起票後の手順を案内する: ① Backlog で起票し採番（例 `BUSSW-1234`）を確認 ② フォルダ名を `BUSSW-AI` → `BUSSW-1234` にリネーム ③ ブランチの `XXXX` を採番に置換。
8. スキップ・宛先付きの確認事項がある場合は、最後に**宛先ごとの確認依頼文**を、そのまま送れる形でコードブロックに入れて提示する（下記「確認依頼文の提示」）。
9. （軽量で進めた場合）最後に「**さらに要件の詳細・実装計画（implementation-plan.md）まで深掘りしますか？**」と **y / n**（小文字）で尋ねる（y＝要件・実装計画まで深掘り / n＝起票だけでよい）。y なら step 5 を実行。n なら軽量の Draft で完了。最初から深掘り指定があった場合はこの問いは不要。

## 質問のしかた

- 選択式の対話質問（AskUserQuestion 等）が使える環境では、それで提示して回答を待つ。
- その機能が無い環境では、同じ質問と選択肢を本文に箇条書きで出して回答を待つ。
- 軽量モードの1回のやり取りは**最大5問**。
- 各質問の選択肢は **3〜5個**にし、**A / B / C** のように記号で選べる形にする。最後は必ず「その他（自由記述）」を入れる。
- 各質問には「**わからない／営業・サポート・顧客に確認が必要**」で**その場で答えずスキップできる選択肢**を必ず用意する。
- スキップされた質問は確定回答にせず、`requirements.md` の「確認したいこと」へ集約する。後で適切な人へまとめて聞ける一覧にする。

## 確認依頼文の提示

スキップ・宛先付きの確認事項があるときは、起票の最後に、**宛先ごとにまとめた確認依頼文**を提示する。

- 宛先ごとに1つの**コードブロック**に入れ、そのままコピーして送れる形にする。
- 文面は**非エンジニアにも分かる平易な言葉**にし、技術用語・コード・ファイル名は入れない。
- 各質問は**A / B / C の選択式**（最大5個、最後は必ず「その他」）にして、記号で答えられるようにする。選択肢が作れない質問だけ自由記述にする。
- 構成は「簡単な前置き → 質問（選択肢付き） → 回答のお願い」。何の件かが一読で分かるようにする。
- 送り先がチャット等で独自記法を要する場合は、その媒体の形式に合わせて整える（特定のツールに依存しない）。

提示する文面の形（宛先ごと）:

```
[機能名]の件で、確認させてください。
記号（A / B / C…）でお答えいただけると幸いです。

Q1. [平易な言葉の質問]
　　A) [選択肢]
　　B) [選択肢]
　　C) その他（ご記入ください）

Q2. [平易な言葉の質問]
　　A) [選択肢]
　　B) その他（ご記入ください）

お手数おかけしますが、よろしくお願いいたします。
```

## 出力先・ブランチ名

- 配置: `.work/backlog/BUSSW-AI/{requirements,implementation-plan}.md`
- ブランチ: `<feat|fix|chore|refactor>/BUSSW-XXXX-<英語 kebab-case・10語以内>`
  - `feat`=新機能 / `fix`=修正 / `chore`=整備 / `refactor`=リファクタリング。
  - `XXXX` は採番後に置き換える（ユーザーからの明示がない場合、生成時は埋めない）。

## ルール（必守）

- `requirements.md` は Backlog 記法で出力する。具体規則（`*` 箇条書き・ネストは半角スペース4つ・表は GFM 等）は `docs/backlog-notation.md` に従う。
- 推測は必ず「前提」欄に置き、確定事項として書かない。
- 軽量モードでコード・DB・技術選定の質問をしない／5問を超えない。
- 必須情報が欠けたまま黙って確定要件として生成しない（壁打ちで補う、または「前提」「確認したいこと」に逃がす）。
- `implementation-plan.md` は、境界をまたぐ場合の契約（§3）と非機能の制約（§4）を埋め、ステップだけの薄い計画にしない。
- `.work/backlog/BUSSW-AI/` 以外に生成しない。

## Examples

- 「請求の画面で、金額をもっと見やすくしてほしい」（軽量）→ 誰の画面か／変更か追加か／何に困っているか等を最大5問で確認し、`requirements.md` 前半を埋めて Draft 起票。推測は「前提」、未確定は「確認したいこと」へ。ブランチ案 `feat/BUSSW-XXXX-highlight-invoice-total`。
- 上記の起票後「要件と計画まで詰めて着手したい」（深掘り・経路 A）→ `requirements.md` 後半・`implementation-plan.md` を本スキルの質問で埋める。
- grill-me 済みのうえで Backlog 起票まで（深掘り・経路 B）→ まとめを template-guide に転記 → 必要なら backlog-markdown-formatting → 配置・ブランチ・起票後手順を案内。
- 「決済処理で NullPointerException が出る不具合を修正」（情報十分）→ 壁打ちは1〜2問に絞ってすぐ Draft 起票。ブランチ案 `fix/BUSSW-XXXX-fix-payment-null-pointer`。
