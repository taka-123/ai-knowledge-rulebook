# template-guide（backlog-intake-planner 出力雛形）

生成物は `requirements.md`（確認サマリー＋要件定義）と `implementation-plan.md`（そのまま実行できる実装計画）の2つ。
この2つを対で読めば、非エンジニアは意図を確認でき、実装者は別資料なしで曖昧さなく完成まで進められる——を満たすことが目的。

---

## ブランチ命名規則

| 種別  | パターン                         | 使いどころ                       |
| ----- | -------------------------------- | -------------------------------- |
| feat  | `feat/BUSSW-XXXX-<description>`  | 新機能・新規開発                 |
| fix   | `fix/BUSSW-XXXX-<description>`   | バグ修正・不具合対応             |
| chore | `chore/BUSSW-XXXX-<description>` | 設定変更・整備・リファクタリング |

- `<description>` は英語の kebab-case で10語以内。
- `XXXX` は Backlog 起票後に採番された番号へ置き換える（生成時は `XXXX` のまま）。

出力先: `.work/backlog/BUSSW-AI/{requirements,implementation-plan}.md`（起票・採番後に `BUSSW-XXXX` へリネーム）。

## requirements.md テンプレート

> 記法: `backlog-notation.md`（同じ docs/ 内）に従う。Backlog 記法で、`*` 箇条書き・ネストは半角スペース4つ・表は GFM。
> 前半は非エンジニアが読んで「意図が正しく汲まれている／欲しいものが完成しそう」と確認できること。後半は実装者・レビュアー向けで技術用語可。

```markdown
# [チケットタイトル]

## 対応

（PR 作成後に記入する）

<!-- ===== 前半: 確認用サマリー（非エンジニア向け・専門用語なし） ===== -->

## 概要

[なぜ・だれが・なにを、を 1〜3 文。読んだ人が「自分の困りごとを正しく理解してくれている」と分かる平易さで]

## 完成イメージ（できるようになること）

* [利用者から見て何ができるようになるか。画面・操作・結果の言葉で]
* [〃]

## やらないこと（スコープ外）

* [今回はやらないこと。期待値のズレを防ぐ]

## 前提（こちらで補った・要確認）

* [推測で補った箇所。確定回答ではない。違えば手戻り]

## 確認したいこと / 未確定

> その場で答えが出ず、誰かに確認が必要な点。**宛先を付けて**まとめて聞ける一覧にする。
* [宛先: 営業・サポート] [質問]
* [宛先: お客さん] [質問]
* [宛先: 社内（エンジニア判断）] [質問]

<!-- ===== 後半: 実装・レビュー用（技術用語可） ===== -->

## 要件（詳細）

* 要件1: [作るものの仕様。入力 / 出力 / 条件まで具体的に]
* 要件2: [〃]

## 受け入れ条件

> PR を作る前に、最低限ここがすべて動作確認できている状態にする（pre-PR ゲート）。各条件の頭に対応する要件番号を付け、確認できた時点でチェックする＝どこまでテストしたかが分かる。

* [ ] [要件1] [ユーザー視点で動作確認できる粒度の条件]
* [ ] [要件2] [〃]
* [ ] [異常系・境界値・権限差・既存影響など横断の観点]

## DB変更（無ければ「なし」と書く）

> テーブルごとにサブ見出しを立て、その下に変更カラムを書く（テーブル名を列に混ぜない）。

### [テーブル名]（新規作成 / カラム追加 など）

| カラム | 型 / 制約 | 内容 |
| --- | --- | --- |
| [カラム] | [型・NULL・DEFAULT] | [用途] |

## 影響範囲・レビュー観点

* 影響リポジトリ: [repo1 / repo2]
* 特に見てほしい観点: [例: 認可漏れ / 既存 API の後方互換 / 計算ロジックの正しさ]

## 用語（非エンジニアに馴染みのない語がある場合のみ。無ければセクションごと削除）

| 用語 | 定義 |
| --- | --- |
| [用語] | [平易な定義] |
```

書き方の指針:

- 前半（概要〜確認したいこと）だけで、非エンジニアが要求の合致を確認できること。
- 後半（要件〜レビュー観点）で、実装者が「何を作るか」、レビュアーが「どこをどの観点で見るか」「どこまでテスト済みか」を掴めること。
- 前半は価値・動作の言葉、後半は仕様の言葉。同じ内容をそのまま二度書かず、粒度を変える。
- 1要件＝1検証単位を目安にする。1つの要件に独立して確認できる事項が複数含まれる、または長文化して読みづらい場合は枝番（要件1-1 / 要件1-2）に分ける。受け入れ条件は対応する要件番号（枝番含む）で必ず紐付ける。
- 専門用語が前半に混ざると非エンジニアが詰まる。混ざる場合は「用語」セクションで補う。

## implementation-plan.md テンプレート

> 記法: 標準 GFM。読者: 実装する AI・エンジニア。
> 役割: requirements.md と対で読めば**別資料なしで、上から実行して完成できる実装計画／スプリント計画**。設計判断（データモデル・契約・制御ルール・制約）を内包し、曖昧なまま実装が進む余地を残さない。

```markdown
# 実装計画: [機能名]

> 関連チケット: BUSSW-XXXX ／ requirements.md と対で読む

## 0. 着手ブロッカー（実装前に確定すること）

> 埋まらないと途中で詰まる。着手前に必ず解消する。無ければ「なし」。

- [ ] [未確定の repo・仕様・依存 ＋ どう確定するか（誰に聞く／何を調べる）]

## 1. 方針

[採用する作り方と理由を数行。なぜ他案でなくこれか]

## 1.5 参考にする既存実装（あれば）
> 似た既存実装があれば、**具体的なファイル・クラス・メソッド名**まで書く。「既存を参考」で止めない。実装者がどこを読めばいいか即分かる状態にする。
- [対象（UI / ロジック等）]: `[ファイルパス]` の `[クラス/メソッド]` — [何を参考にするか]

## 2. データモデル（DB変更がある場合のみ）

### テーブル: [名前]

| カラム | 型         | NULL | 説明   |
| ------ | ---------- | ---- | ------ |
| id     | biginteger | 不可 | 主キー |

- 制約・インデックス・監査カラム（created/modified 等）を明記

## 3. インターフェース／契約（境界をまたぐ場合のみ）

> API・画面とサーバ間・サービス間・関数の入出力など、実装が依存する契約を確定させる。

### API / エンドポイント

| エンドポイント | メソッド | リクエスト | レスポンス | 認可   |
| -------------- | -------- | ---------- | ---------- | ------ |
| `/api/[path]`  | POST     | [項目]     | [項目・形] | [条件] |

### 関数 / イベント（該当時）

- [シグネチャ・入出力・前提]

## 4. 制御ルール・エッジケース・制約（曖昧さの排除）

- 制御ルール: [冪等性 / 計算ロジック / 状態遷移など、実装中に迷う分岐]
- エッジケース: [0 件・大量データ・同時実行・権限差]
- 制約: [セキュリティ（認可・データ保護） / 性能（N+1・インデックス） / 後方互換（既存への影響）]

## 5. 実装ステップ（スプリント計画）

> このまま上から実行すれば完成する粒度。各ステップに「対象・やること・完了の確認方法」を必ず付け、§2〜4 を参照させる。

| # | 対象（repo / 領域） | やること | 完了の確認方法 | 依存 |
| --- | --- | --- | --- | --- |
| 1 | [repo] | [作業（§2 のテーブル等を参照）] | [検証コマンド or 観察できる結果] | - |
| 2 | [repo] | [作業（§3 の契約に沿う）] | [〃] | 1 |

## 6. 検証ゲート（完了判定）

- [ ] lint / build が通る
- [ ] requirements.md の受け入れ条件をすべて満たす
- [ ] （危険領域）reviewer の Verdict: ready
- [ ] （schema 変更）`sc-ddl-migration-impact` の結果を添付

## 7. 見積もり

| ステップ | 工数       |
| -------- | ---------- |
| **合計** | **[合計]** |
```

書き方の指針:

- §2〜4（データモデル・契約・制御ルール・制約）で**実装判断に必要な設計を確定**し、§5 の各ステップからそれらを参照させる。これにより設計書を別に持たなくても実装が進む。
- 「実装ステップ」は、AI が読んでそのまま着手でき、各ステップの完了を自分で判定できる粒度にする（対象・やること・完了確認の3点を欠かさない）。
- §3 契約・§4 制約が薄いと、入出力や非機能の判断が現場任せになり、適当な実装を招く。境界をまたぐ・非機能要件がある場合は必ず埋める。
