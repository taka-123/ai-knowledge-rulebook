# diff-quality-gate

未コミット差分（staged 含む）**全体**の品質ゲート。指摘のみ。**コミット・amend・push はしない**。
本スキルは**静的レビューのみ**（差分の読取）。lint / 型チェック / テスト実行は行わない（別途依頼）。

文書ファイル（`*.md` 等）の本文品質は、差分に含まれる範囲で **`document-authoring` の SKILL.md を Read して準用**する（ルールをここに再掲しない）。

## Procedure

1. `git status` / `git diff`（staged・unstaged）と、必要なら untracked の内容を取る。差分が無ければ「差分なし」と報告して終了する。直前作業だけに絞らない。
2. 下記チェックリストで不合格を列挙する。
3. 結果を返す:
   - **合格**: 2〜4 行（気になったが許容した点があれば 1 行）。
   - **不合格**: 指摘を箇条書き（場所・何が問題・直すなら最小案）。即直せそうで依頼範囲内なら修正案まで出してよいが、勝手にコミットしない。機密は**値を出力せず**、ファイル・おおよその箇所と種類だけ示す。

## チェックリスト

いずれも「差分に現れているもの」が対象。リポジトリ全体の健康診断ではない。

### 1. 残骸・一時物

- 会話・試行の経緯だけが残るコメント（「いったん」「後で戻す」「前回は」等、後から読んでも設計意図にならないもの）
- デバッグ用の `console.log` / `var_dump` / 捨てた分岐・コメントアウト塊・未使用 import・到達不能コード
- 実験用フラグ、ハードコードした一時 URL / マジック値、本番に載せる前提でないマーカー

### 2. 説明の質（コメント・名前）

- 今の実装と矛盾する注釈
- 「何をしているか」の復唱だけで、後から効く「なぜ」がない長いコメント（有用な why / 制約は残してよい）
- 周辺の命名・用語と明らかに齟齬する識別子（意図的な場合は、差分か短いコメントから意図が読み取れること）

### 3. 重複と再利用（差分最小とのバランス）

- **不合格寄り**: 同じファイルや近い層に、呼べば足りる既存関数・部品があるのに、同趣旨を新規で書き直している
- **不合格寄り**: コピー増殖で、片方だけ直すと壊れる形になっている
- **許容**: 共通化すると逆に行数・結合・読みにくさが増える局所の重複、モック固有の意図的な複製、1 回きりの使い捨てコードでの割り切り

判断に迷ったら「既存を使った方が読者・将来の修正が明らかに楽か？」を基準にする。楽でないなら共通化圧力をかけない。

### 4. 範囲と意図

- 依頼・作業意図と無関係なリファクタ、フォーマットのみの大量書き換え、依存・設定のついで変更
- 1 つの差分に、説明なく別意図が混在（分割を勧める。メッセージ作業は `commit-message-suggester` / `diff-review-commit` 側）
- 途中までなのに完了に見える差分（呼び出し元の一部だけ変更済み、新旧スタイルが混在したまま等）

### 5. 安全・機密（浅いゲート）

- `.env`、鍵、トークン、個人情報らしい生値が差分に含まれている → **即不合格**（値は出さず、除外・伏せ方だけ示す）
- 深い脆弱性レビューは本 Skill の範囲外（専用の security レビューへ）

### 6. 差分そのものの明らかな欠陥

- 読み取りだけで分かる null 参照・条件逆転・消し残りによる参照切れ
- テストやモックを「通すため」に検証を弱めている変更

## 合格ライン

- 上記の不合格がなく、または指摘が「任意の好み」レベルで、差分最小・既存踏襲の説明がつく
- 機密混入がない
