# verify-grill

grill-me が**要件**を固めた後、**検証**を先に設計する。実装前に使う。

否定テスト必須の詳細は **`TEST_CONTRACT.md` が正本**。本スキルは設計手順のみ。
前提: 対象プロジェクトに `TEST_CONTRACT.md` が存在すること。無い場合は本スキルを適用しない。

## 目的

チェックリストの最小充足ではなく、**要件の一文一文を検証項目に変換**する。

## 手順

### 1. 要件を機械的に分解

一次情報（SPEC / チケット / 合意済み要件）から列挙する。

```
[ID] 要件引用（そのまま）
  肯定: ユーザーが〇〇すると △△ になる
  否定: ユーザーが×× しても □□ にならない（状態不変・禁止 UI なし 等）
  不変条件: 常に守るべき制約
  観測: DOM / API / ログ / スクリーンショット（可聴は自動化不能へ）
```

**否定が 1 つも出ない機能は不合格** — ただし **共通レイヤーの E2E で担保済み**なら、設計書に参照テスト名を書けば可（`TEST_CONTRACT.md`「共通レイヤー参照」）。

例（汎用）:

- 権限 → 「権限のないユーザーが実行しても拒否され、状態が変わらない」
- バリデーション → 「不正入力で成功扱いにならない」
- 冪等性 → 「同一操作の再実行で重複作成・二重送信にならない」

### 2. TEST_CONTRACT と突合

`TEST_CONTRACT.md` の S1–S8・実装容易性義務・自動化不能欄を確認。

### 3. 検証設計書を出力

```markdown
## [機能名] 検証設計（verify-grill 出力）

### 肯定
1. 名前 — 手順 — 期待 — 確認方法（自動 / 再現 / 目視）

### 否定（必須、または共通レイヤー参照）
1. 名前 — 手順 — 期待 — 要件引用（参照の場合は `→ [テスト名]`）

### 不変条件
1.

### 禁止するソルバー / ショートカット
-

### 自動化不能（可聴等）
-

### 未決（ユーザー / DECISIONS 候補）
-
```

### 4. 実装指示は書かない

本スキルは**検証設計のみ**。アプリコード・テストファイルは**まだ書かない**。

## 完了条件

- [ ] 肯定 ≥ 1（主要パス）
- [ ] 否定 ≥ 1、または共通レイヤー参照が明記されている
- [ ] 禁止ソルバーが明示されている
- [ ] spec-auditor が TRACEABILITY を生成できる粒度

## 禁止

- 実装の都合で要件を弱める（必要なら停止条件としてユーザーへ）
- 「正解データを読んで通す」だけの設計（S1 条件なし）
