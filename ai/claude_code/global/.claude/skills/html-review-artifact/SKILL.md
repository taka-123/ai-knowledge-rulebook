# html-review-artifact

人間が一度で判断できるレビュー用 HTML を生成する。
デフォルトは **HTML を作らない**。HTML が効く成果物かを先に判定し、Yes の場合のみ self-contained な単一 HTML を生成する。

chatwork-formatter と同型の **種別判定 → 形式選択** が核。
chatwork-formatter との違い: 本 Skill はレビュー用に **構造を再構成すること自体が責務**。
ただし再構成してよいのは **提示順序・表・折りたたみ** などの構造のみで、**事実・論点・項目数は増幅しない**。

## Procedure

1. **判定**: 下記「Gate criteria」に従い、HTML を作るか決める。No なら Markdown（または JSON / Schema）のままにし、理由を 1 行添えて終了する。
2. **正本の確認**: 正本 Markdown があるか確認する。ある → HTML は生成物。ない → 使い捨てレビューなら HTML のみ可。継続更新される文書なら正本 MD の新設を先に提案する。
3. **生成**: 単一 self-contained HTML。下記「HTML generation rules」と「Restructure boundary」を守る。
4. **配置と提示**: 保存先は下記「Output destination」。ブラウザで開く導線を示す。`docs/` や正本 MD と同じディレクトリに HTML を置かない。

## Gate criteria

デフォルトは **No**（HTML を作らない）。次の除外条件に当てはまれば、他の条件に関わらず **No**。

| 除外（即 No）     | 典型例                                       | 代替                                      |
| ----------------- | -------------------------------------------- | ----------------------------------------- |
| Git diff が主目的 | CLAUDE.md、AGENTS.md、ADR、実装計画、PR 説明 | Markdown のまま                           |
| 機械処理が主目的  | テストケース一覧、API 契約、設定スキーマ     | JSON / JSON Schema                        |
| 交流・会話系      | 返信、依頼、確認、お礼、短い CW 返信         | chatwork-formatter またはプレーンテキスト |

除外に当てはまらない場合、次の **即 Yes** 条件のいずれか 1 つでも該当すれば HTML を作る。
「一応 2 案ある」程度の弱い比較は即 Yes にしない。意思決定の主目的が比較である場合のみ。

| 即 Yes                         | 理由                                     |
| ------------------------------ | ---------------------------------------- |
| 複数案の比較が意思決定の主目的 | 横並び比較・表がレビュー精度を上げる     |
| 操作して決める必要がある       | フィルタ・タブ・折りたたみ等が判断に必要 |

即 Yes にも該当しない場合、次の 3 問のうち **2 つ以上 Yes** なら HTML を作る。

| 問い                                                 | Yes なら  |
| ---------------------------------------------------- | --------- |
| 図、表、フロー、画面遷移がレビュー理解に不可欠か     | HTML 候補 |
| 非エンジニアも読むか                                 | HTML 候補 |
| 読むだけでは足りないか（操作・折りたたみ等が必要か） | HTML 候補 |

単なる箇条書きや週次進捗の列挙だけでは、上記 3 問は No とする（Markdown で足りる）。

それでも迷ったら **No**（デフォルトは HTML を作らない）。

## Restructure boundary

| してよい（構造）                               | してはいけない（内容）                      |
| ---------------------------------------------- | ------------------------------------------- |
| 結論・推奨案を先頭に移動                       | 論点・項目数を増やす（2 案を 3 案にしない） |
| 比較軸を `table` に整理                        | 根拠のない事実・数値を追加                  |
| 未決事項・前提・根拠を独立セクションに分離     | 元にないリスクや推奨を捏造                  |
| 長文を `details` / `summary` で畳む            | 元文にない主張・結論を追加                  |
| 意味的 HTML（`section`, `h1`〜`h3`, `figure`） | 元文の意味を変える要約・言い換え            |

## HTML generation rules

目的: 読み手が一度で判断できるレビュー資料にする。

必須:

- 最初に結論と推奨案を置く
- 比較がある場合は比較軸を `table` にする
- 重要な論点は図またはフローで示す（必要時のみ。SVG インライン可）
- 未決事項を独立したセクションにする
- 根拠と前提を分離する
- 見出しは `h1` → `h2` → `h3` の順
- コードは必要最小限だけ載せる
- 長い説明は `details` / `summary` で畳む
- CSS はインラインまたは `<style>` で完結（外部 CDN 依存を避ける）

避ける:

- 目的のないアニメーション
- 装飾のためだけのカード乱用
- 絵文字の乱用
- CSS だけで意味を伝える構造
- 外部依存が多い構成
- レイアウト専用の `div` の深い入れ子
- ソースコード全量の貼り付け

## Output destination

本 Skill の価値は **ブラウザで開いてレビューできること**。HTML は必ずファイルとして保存し、開く導線を示す。会話内への HTML 全文出力は行わない。

優先順:

1. ユーザーが保存先を明示 → そのパス（正本ディレクトリ `docs/` 等は避ける）
2. プロジェクトに `artifacts/` がある → `artifacts/<内容を表すスラッグ>.html`
3. 上記がなければ一時ディレクトリへ保存（例: `/tmp/<スラッグ>.html`）し、ブラウザで開くコマンドを提示（macOS: `open <path>`）

`artifacts/` は生成物のため、原則 `.gitignore` 対象とする。チームでリポジトリ内共有する場合はプロジェクト方針に従う。

## Rules

- HTML を正本にしない。継続更新する文書は Markdown（または構造化データ）を正本とする。
- 恒久ルール（`CLAUDE.md`、`AGENTS.md`、`.cursor/rules` 等）を HTML 化しない。
- 交流文・チャット返信を HTML 化しない。
- 事実を捏造しない。不明点は未決事項に残す。

## Examples

### Example 1: 2 案比較のレビュー（即 Yes → HTML 生成）

Input: 「認証基盤移行の一括移行 vs 段階移行を、PM がブラウザでレビューできる形にして」

Output:

- 判定: 複数案の比較が意思決定の主目的（即 Yes）
- `artifacts/auth-migration-comparison.html` を生成し `open` で開く（`artifacts/` がない場合は `/tmp/` へ保存）
- HTML 構成: 冒頭に結論・推奨案 → 比較 `table`（案 / 利点 / リスク / 移行期間）→ 未決事項セクション

### Example 2: 恒久ルールの HTML 化依頼（除外 → 拒否）

Input: 「`CLAUDE.md` を HTML にして見やすくして」

Output:

- 判定: Git diff が主目的（除外・即 No）
- HTML は作らない
- 理由を 1 行: 「恒久ルールは Markdown 正本のまま維持し、Git diff で追うべきです」
- 代替: 必要なら特定セクションの要約を会話内 Markdown で提示

### Example 3: 調査結果の PM 共有（2/3 問 Yes → HTML ＋ 二層構成）

Input: 「API 障害の調査結果を PM に共有したい。`docs/incidents/2026-07-api.md` が正本」

Output:

- 判定: 非エンジニアが読む ＋ 時系列・影響範囲の表がレビュー理解に不可欠（2/3 問 Yes）
- 正本 `docs/incidents/2026-07-api.md` は編集しない
- `artifacts/incident-2026-07-api-review.html` を生成
- HTML 構成: 結論（影響範囲・現在ステータス）→ 時系列 `table` → 根本原因 → 対応済み / 未決事項
- 完了報告で正本 MD と HTML の役割分担を明示

### Example 4: テストケース一覧の HTML 化（除外 → JSON 提案）

Input: 「テストケース一覧を HTML で出力して」

Output:

- 判定: 機械処理が主目的（除外・即 No）
- HTML は作らない
- 代替: JSON または JSON Schema での出力を提案（後続の自動処理・CI 連携向け）
- 人間向けの読み物が別途必要なら、その要件（比較・非エンジニア向け等）を確認してから再判定
