# html-review-artifact

人間が一度で判断できるレビュー用 HTML を生成する。
AI の長い調査・比較アウトプットは、人間が読み解いて再指示するまでのタイムラグがボトルネックになる。
HTML が効く成果物なら明示依頼を待たずに HTML 化し、効かない成果物には作らない。

再構成してよいのは **提示順序・表・折りたたみ** などの構造のみで、**事実・論点・項目数は増幅しない**。

## Procedure

1. **経路判定**:
   - いずれの経路でも、まず Gate criteria の「除外（即 No）」を確認する。該当すれば HTML を作らず代替を提示して終了する（正本の置き換え・機械処理・交流・会話系を含む）。
   - **明示依頼あり**（「HTMLで」「ブラウザで見たい」等）→ 除外に該当しなければ生成する。
   - **明示依頼なし** → 除外後に下記 Gate criteria の Yes 判定を行う。Yes なら生成する（提案で止めない。使い捨て生成物なのでコストは低い）。No なら通常の形式のままにする。
2. **正本の確認**: 正本 Markdown があるか確認する。ある → HTML は生成物。ない → 使い捨てレビューなら HTML のみ可。継続更新される文書なら正本 MD の新設を先に提案する。
3. **生成**: 単一 self-contained HTML。本 Skill 同梱の `assets/template.html` を読んでから、下記「HTML generation rules」と「Restructure boundary」に従って組む。
4. **配置と提示**: 保存先は下記「Output destination」。ブラウザで開く導線を示す。`docs/` や正本 MD と同じディレクトリに HTML を置かない。

## Gate criteria（明示依頼がない場合の自発判定）

デフォルトは **No**（HTML を作らない）。次の除外条件に当てはまれば、他の条件に関わらず **No**。

| 除外（即 No）              | 典型例                                   | 代替                                      |
| -------------------------- | ---------------------------------------- | ----------------------------------------- |
| 正本の作成・置き換えが目的 | ADR・実装計画・PR 説明を HTML で書く     | 正本は Markdown。理解用ビューは別途判定   |
| 機械処理が主目的           | テストケース一覧、API 契約、設定スキーマ | JSON / JSON Schema                        |
| 交流・会話系               | 返信、依頼、確認、お礼、短い CW 返信     | chatwork-formatter またはプレーンテキスト |

除外に当てはまらない場合、次の **即 Yes** 条件のいずれか 1 つでも該当すれば HTML を作る。
「一応 2 案ある」程度の弱い比較は即 Yes にしない。意思決定の主目的が比較である場合のみ。

| 即 Yes                                 | 理由                                          |
| -------------------------------------- | --------------------------------------------- |
| 複数案の比較が意思決定の主目的         | 横並び比較・表がレビュー精度を上げる          |
| 図・フロー・時系列なしでは判断できない | `flow` / `timeline` / 色分け `table` が不可欠 |

即 Yes にも該当しない場合、次の 3 問のうち **2 つ以上 Yes** なら HTML を作る。

| 問い                                                   | Yes なら  |
| ------------------------------------------------------ | --------- |
| 図、表、フロー、画面遷移がレビュー理解を大きく助けるか | HTML 候補 |
| 非エンジニアも読むか                                   | HTML 候補 |
| 長文で、結論先出し＋詳細折りたたみの二層構成が効くか   | HTML 候補 |

単なる箇条書きや週次進捗の列挙だけでは、上記 3 問は No とする（Markdown で足りる）。

それでも迷ったら **No**（自発生成のデフォルトは No）。

## Restructure boundary

- してよい（構造の再構成）: 結論・推奨案の先頭移動、比較軸の `table` 整理、未決事項・前提・根拠の独立セクション分離、長文の `details` / `summary` 折りたたみ、意味的 HTML（`section`, `h1`〜`h3`, `figure`）。
- してはいけない（内容の増幅・改変）: 論点・項目数を増やす（2 案を 3 案にしない）、元文にない事実・主張・結論の追加、元文の意味を変える要約・言い換え。

## HTML generation rules

目的: **最初の画面で結論と重大度が判断できる**こと。見た目は `html-web-report` と共通テンプレ。用途は意思決定。

- `assets/template.html` の `<style>` は無変更で全文コピーし、`.shell` 内だけを組み替える。ライト固定（テーマ切替・JS なし）。色はトークン（ok / warn / danger / accent）経由のみ。色付き面（`.verdict` 等）に置く `code` はテンプレの規則に従う。
- 冒頭は `.hero`: 種別 `.kicker`・`h1`・`.verdict`（結論 1〜2 文、色付き面＋左罫）を必須で入れる。`.facts` は判断に効く数値・条件があるときだけ（無ければ省略。埋め草や元文にない指標を作らない）。`.contents` はセクションが複数あるとき任意で付ける。
- 使う／使わないが明確なら `.split` + `.panel`（ok・✓ / warn・✕）をヒーロー直下に置く。
- 情報の型ごとにコンポーネントを対応させる:

| 情報の型               | コンポーネント                                              |
| ---------------------- | ----------------------------------------------------------- |
| ヒーロー（種別・結論） | `.hero` / `.kicker` / `.verdict` / `.facts` / `.contents`   |
| 使う／使わない         | `.split` + `.panel`（ok / warn）                            |
| 状態・判定・重大度     | `badge` ＋ 行に色付けした `table`                           |
| 比較                   | 比較 `table`（比較軸を行に）                                |
| 分岐のある手順         | `flow`（分岐は `flow-branch` で並列に描く。直列に潰さない） |
| 時系列・経過           | `timeline`（時刻は `.time`、重大イベントは li に色クラス）  |
| 役割分担・案の対置     | `cols` ＋ `card`（推奨案は `card recommended`）             |
| 警告・禁止事項         | `callout`（danger / warn）                                  |
| 長文・コマンド列・補足 | `details` で畳む（既定は閉じる）                            |

- フロー図を SVG の座標手書きで作らない。template の `flow` を使う。
- 未決事項は独立セクション（無ければ「未決事項なし」）。根拠と前提は本文と分離する。
- 正本 MD が無い使い捨てでは、ヘッダー・フッターの「正本」chip を削除する。
- コードは判断に必要な行だけ。色・バッジは状態・重大度・順序の符号化のみ。
- 提出前: (1) 最初の画面だけで結論と重大度が判断できるか。(2) トークン外の色が混入していないか。

## Output destination

本 Skill の価値は **ブラウザで開いてレビューできること**。HTML は必ずファイルとして保存し、開く導線を示す。会話内への HTML 全文出力は行わない。

優先順:

1. ユーザーが保存先を明示 → そのパス（正本ディレクトリ `docs/` 等は避ける）
2. 進行中タスクの作業ディレクトリ（例: `.work/backlog/<チケット>/`）があればその直下
3. 作業ディレクトリ直下の `.work/artifacts/<内容を表すスラッグ>.html`（ディレクトリが無ければ作成する）

保存後、ブラウザで開くコマンドを提示する（macOS: `open <path>`）。
生成 HTML は使い捨ての生成物のため、原則 `.gitignore` 対象とする。チームでリポジトリ内共有する場合はプロジェクト方針に従う。

## Examples

### Example 1: 2 案比較のレビュー（明示依頼 → ゲート不要で生成）

Input: 「認証基盤移行の一括移行 vs 段階移行を、PM がブラウザでレビューできる形にして」

Output:

- 経路: 明示依頼（ブラウザでレビュー）→ 除外に該当せず生成
- `.work/backlog/<チケット>/auth-migration-comparison.html` を生成し `open` コマンドを提示
- HTML 構成: 冒頭に結論・推奨案 → 比較 `table`（案 / 利点 / リスク / 移行期間）→ 未決事項セクション

### Example 2: 週次進捗の共有（明示依頼なし・Gate No → Markdown のまま）

Input: 「今週の進捗をまとめてチームに共有して」

Output:

- 経路: 明示依頼なし → Gate criteria で判定
- 判定: 進捗項目の列挙のみ。比較・図表・二層構成は不要（即 Yes なし、3 問すべて No）
- HTML は作らず Markdown で提示。ChatWork 宛なら chatwork-formatter

### Example 3: 調査結果の PM 共有（明示依頼なし・Gate Yes → 自発生成）

Input: 「API 障害の調査結果を PM に共有したい。`docs/incidents/2026-07-api.md` が正本」

Output:

- 経路: 明示依頼なし → Gate criteria で判定
- 判定: 非エンジニアが読む ＋ 時系列・影響範囲の図表が理解を大きく助ける（2/3 問 Yes）→ 生成
- 正本 `docs/incidents/2026-07-api.md` は編集せず、`.work/artifacts/incident-2026-07-api-review.html` を生成
- HTML 構成: 結論（影響範囲・現在ステータス）→ `timeline` → 根本原因 → 対応済み / 未決事項
- 完了報告で正本 MD と HTML の役割分担を明示

### Example 4: テストケース一覧の HTML 化（明示依頼だが機械処理目的 → 除外で拒否）

Input: 「テストケース一覧を HTML で出力して」

Output:

- 経路: 明示依頼あり → 除外（機械処理が主目的）に該当
- HTML は作らず、JSON または JSON Schema での出力を提案する
- 人間向けの読み物が別途必要なら、その要件（比較・非エンジニア向け等）を確認してから再判定する
