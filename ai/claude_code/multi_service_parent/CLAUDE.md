# CLAUDE.md (Parent Workspace)

このファイルは、複数リポジトリをまとめた親ディレクトリで Claude Code が作業する際のワークスペース規約です。

実装規約は子リポジトリの `CLAUDE.md` を正とする。本ファイルは横断作業の索引・調停・整合性確認の起点に限り、実装の詳細は書かない。

## 位置づけ

- 適用順序: ユーザーが明示したパスのルール > 子 repo の `CLAUDE.md` > 本ファイル > グローバル `~/.claude/CLAUDE.md`。
- 競合時は、作業パスに近いほうを優先する。
- リポジトリ名やディレクトリ名だけで責務を決めつけず、実ファイル、設定、依存関係、README、子 repo の規約から実態を確認する。

## 子リポジトリの解決

- ユーザーが子リポジトリ名、サービス名、機能名、パスを言及したら、最初に該当 repo の `CLAUDE.md` を Read する。
- 影響範囲が不明な場合は、最小限の探索で候補 repo を特定し、推定であることを明記する。
- 子 repo の規約と本ファイルが競合した場合は、子 repo を優先する。
- 子 repo に `CLAUDE.md` がない場合は、README、設定、テスト、CI、既存コードから規約を推定し、推定であることを明記する。

## 横断タスク契約

複数 repo にまたがる非自明な変更では、実装前に短く整理する。

1. Goal — 何を達成するか
2. Affected repos — 影響する repo と理由
3. Boundaries — API、DB、schema、認証、イベント、環境変数などの契約境界
4. Change order — 変更順序、互換性維持、移行手順
5. Done when — 各 repo および統合観点で何が通れば完了か

誤った repo 選定や変更順序のコストが高い場合は確認する。そうでなければ、前提を明記して進める。

## 横断変更の原則

- 変更は、依頼を満たす最小の repo・最小のファイル範囲に留める。
- 一方の repo だけを変更して、API、schema、型、環境変数、認証、権限、migration、CI との整合性を崩さない。
- 互換性破壊変更は、段階移行、後方互換、ロールバック方法を検討する。
- 横断的なリファクタ、構成変更、依存更新、CI/CD 変更は、明示依頼がない限り提案止まりにする。

## 並行探索

- 横断調査が重い場合は、Subagent / Task ツールで repo ごとに並行探索する。
- メインのコンテキストには集約結果のみ取り込む。
- 横断ルールが複数ある場合は `.claude/rules/<topic>.md` に分離し、本ファイル本体を肥大化させない。
